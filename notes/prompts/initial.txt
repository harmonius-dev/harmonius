This library is an ambitious render graph project to build a production-grade
cross-platform renderer for high performance desktop games. The user-facing API
is 100% safe and no unsafe operations run on user threads. Separate background
render and IO threads are used to execute unsafe C++ code that implements an
extremely high performance backend. This allows the user to think and debug at a
higher level, while delegating the low level work to the platform-specific
backend implementation. Focus on supporting the best API for each target desktop
platform. This render graph is designed for 100% GPU driven renderers.
CPU-driven renderers are not supported and not a priority.

The render graph should be split into a building phase and an execution phase.
During render graph building, an entirely safe, declarative description of the
render graph is defined that describes how rendering should function for each
application. The graph does not change frame-to-frame, but includes
functionality that allows it to cull unnecessary passes when certain features
are unnecessary, disabled, or don't fit in the GPU's performance budget.

The render graph is compiled into an execution plan. The execution plan is an
optimized index of the render graph. The execution plan specializes the render
graph for the runtime host, disabling any unsupported features in the render
graph. The execution plan will need to disable whether each node in the graph
has the required features to function on the runtime system, and if not, it will
disable and cull those nodes. This allows the execution plan to be extremely
efficient when executed the render loop. The render loop may perform additional
graph node culling if more features are disabled at runtime, for example, to
improve performance when the framerate drops too low. Graph nodes are disabled
transitively and are dependencies are automatically disabled when they are
unused. Design the most efficient execution without performing any wasted work.

The graph executor is responsible for taking an execution plan and performing it
each frame, updating any dynamic frame-specific state. The executor will need to
manage many moving parts. It needs to orchestrate loading GPU buffer and texture
data asynchronously across threads, and encoding platform-specific render
commands on dedicated threads across platforms with high performance.

Transparent objects will need to be sorted by distance. To support this, we need
the CPU to sort the data before uploading it to the GPU in some cases. Make sure
this is generic and not specific to transparency. This needs to be done
dynamically because objects and perspectives frequently change in games.

Similar to Unreal Engine, the executor uses a pool of background workers to load
textures and buffers from the disk using their own transfer queue pool, so the
render threads do not get blocked waiting for disk I/O or copies from CPU to
GPU. The background workers use one or more transfer queues to read and load
content into the GPU buffers asynchronously, using an API like
MTLIOCommandBuffer and DirectStorage. This means they may be reading from disk
and will most likely be IO bound, not CPU bound. They will interact with the GPU
mainly just to synchronize buffer and texture access.

The render threads will encode render commands into a command buffer, then
dispatch it to the queue, synchronizing with it and pacing the render loop to
match the display refresh rate and performance. Since different frames can be
dispatched on separate render thread queues, we will need to be careful about
assets that are shared between render threads (for example, when rendering to
separate windows or for split-screen). This ensures that any dynamic resources
are synchronized between queues, and two writes do not occur unnecessarily. This
requires careful execution planning. Since a separate render thread drives each
visible camera, this allows for increased performance at the cost of
synchronization.

Another important aspect will be support for asynchronous compute. These GPU
operations take more than one frame to complete, so they must be carefully
managed. This should be integrated with the IO module so that compute shaders
can process loaded geometry to produce new assets in real-time. Support the
generation of assets using mesh shaders, as well as readback of those outputs so
that generated assets can be persistent. This render graph should be very useful
for generating 3D terrains and infinite voxel worlds on the GPU. Mesh shaders
replace the traditional geometry pipeline entirely in this render graph.
