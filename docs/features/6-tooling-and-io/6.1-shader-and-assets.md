# 6.1 Shader and Asset Pipeline

## F-6.1.1 Data-Driven Shader Graph Authoring

Visual shader graphs defined as DAGs of typed nodes connected by typed edges. Graphs are serialized to a human-readable textual format for version control diffability and merge conflict resolution. Each graph can declare permutation axes for compile-time branching (e.g., alpha mode, quality tier).

- **Requirements:** [R-3.3.8](../../requirements/3-nonfunctional/3.3-data-constraints.md) single shader IR, [R-3.3.9](../../requirements/3-nonfunctional/3.3-data-constraints.md) textual format

## F-6.1.2 Shader Graph Compilation

A standalone compiler transforms shader graphs into platform-native shader code via a platform-neutral intermediate representation. Supports validation (type checking, cycle detection), permutation enumeration, reflection data extraction, and cross-compilation to all target backends.

- **Requirements:** [R-3.3.8](../../requirements/3-nonfunctional/3.3-data-constraints.md) single shader IR

## F-6.1.3 Custom Shader Nodes

Third-party shader node types can be registered via a node registry, providing a slot signature and IR lowering function. This enables engine-specific or game-specific shader operations without modifying the compiler.

## F-6.1.4 Resource CPU Readback

Typed asynchronous readback of GPU buffer data to CPU memory via staging buffers and fence synchronization. Used for debugging, screenshots, procedural data persistence, and physics feedback.

- **Requirements:** [R-3.2.4](../../requirements/3-nonfunctional/3.2-hardware.md) transfer queues

## F-6.1.5 Asset Import Pipeline

Source assets from Blender, Maya, Photoshop, and Substance are automatically exported to interchange formats (glTF 2.0,
Alembic, PNG, EXR) via a file watcher daemon that invokes each tool's headless CLI whenever source files change.
Interchange assets are parsed using low-level libraries (cgltf, libalembic, libpng, OpenEXR), optimized and meshletized
via meshoptimizer, compressed with Draco (geometry) and zstd (chunks), and packed into platform-specific zero-copy
binary asset bundles. Bundles are compressed with zstd during transit and storage; at startup they are decompressed on
the CPU and cached locally so that subsequent loads use zero-copy memory-mapped reads with no runtime decompression
overhead. glTF is used as an interchange format after meshes are optimized, not as a raw import format.

- **Requirements:** [R-3.3.10](../../requirements/3-nonfunctional/3.3-data-constraints.md) glTF import format

## F-6.1.6 Domain-Specific Shader Language

A high-level DSL wrapper around the target shading language (HLSL/MSL/GLSL) enabling flexible, modular material and feature composition. The DSL provides composable feature blocks, automatic permutation management, and cross-game reuse of shader logic without sacrificing performance. Compiles down to optimal platform-native shader code through the existing shader compilation pipeline.

## F-6.1.7 Neural Texture Compression

ML-based texture compression that uses small neural networks to decode texture data at runtime, achieving higher compression ratios than traditional block compression formats. Textures are encoded offline into network weights; the runtime decoder evaluates the network per texel in the fragment shader via cooperative vector operations.

## F-6.1.8 Neural Materials

Neural network-based material evaluation where complex multi-layer material stacks are distilled into compact neural representations. The neural material is evaluated in the fragment shader using hardware-accelerated cooperative vectors, replacing expensive conventional material graph evaluation with a single network inference pass.
