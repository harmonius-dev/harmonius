# 3.1 Meshlet Pipeline

## F-3.1.1 Meshlet-Based Virtualized Geometry

All geometry is decomposed into meshlets (small clusters of ~128 vertices and ~128 triangles) stored in a hierarchical cluster structure. Meshlets are the atomic unit of GPU culling, LOD selection, and streaming. This enables fine-grained visibility determination and bandwidth-efficient rendering.

- **Requirements:** [R-1.1.3](../../requirements/1-architecture/1.1-core-constraints.md) mesh shaders, [R-3.2.2](../../requirements/3-nonfunctional/3.2-hardware.md) mesh shaders required, [R-3.2.1](../../requirements/3-nonfunctional/3.2-hardware.md) bindless

## F-3.1.2 Mesh Shader Pipeline (Task/Mesh)

Task (object) shaders perform per-meshlet culling and LOD selection. Surviving meshlets are dispatched to mesh shaders that output triangles directly to the rasterizer. Fragment shaders produce final pixel output. No traditional vertex/index pipeline is used.

- **Requirements:** [R-1.1.3](../../requirements/1-architecture/1.1-core-constraints.md) mesh shaders, [R-1.1.7](../../requirements/1-architecture/1.1-core-constraints.md) no legacy pipelines

## F-3.1.3 GPU-Evaluated Spline Paths

Catmull-Rom / cubic Bezier splines evaluated in the task shader. The spline can be extruded into tube geometry (roads, rails, pipes) rendered via the mesh shader with proper UV mapping and LOD.

- **Requirements:** [R-1.1.3](../../requirements/1-architecture/1.1-core-constraints.md) mesh shaders

## F-3.1.4 Dynamic Tessellation and Displacement

Runtime adaptive tessellation generating triangles to match displacement detail at the current pixel density. Displacement maps or procedural materials drive vertex offsets on dynamically generated sub-triangles, adding geometric detail beyond the base mesh without offline baking.

- **Requirements:** [R-1.1.3](../../requirements/1-architecture/1.1-core-constraints.md) mesh shaders

## F-3.1.5 Visibility Buffer Rendering

64-bit visibility buffer rasterization storing triangle and instance IDs per pixel via atomic operations. Material evaluation is deferred to a fullscreen pass that reads only the visible pixels, eliminating redundant shading and reducing bandwidth versus a traditional G-buffer for high-polygon scenes.

- **Requirements:** [R-1.1.3](../../requirements/1-architecture/1.1-core-constraints.md) mesh shaders, [R-3.2.1](../../requirements/3-nonfunctional/3.2-hardware.md) bindless

## F-3.1.6 Variable Rate Shading (VRS)

Compute-shader-driven variable rate shading that reduces fragment shading rate in perceptually less important screen regions. A per-tile shading rate image is generated based on luminance variance, motion, and edge detection, then applied during rasterization to trade shading quality for performance.

## F-3.1.7 Meshlet Compression (DGF)

Lossy compression format for meshlet geometry data that reduces the storage overhead of meshlet-based virtual geometry systems. Per-meshlet quantization of vertex positions, normals, and UVs with configurable precision, supporting arbitrary topologies. Decoded in the mesh shader at runtime with minimal ALU overhead.

## F-3.1.8 GPU Work Graphs

GPU-self-scheduled work dispatch without host CPU involvement. Producer passes emit variable-count work items consumed by downstream passes, enabling recursive expansion/compaction patterns, adaptive tessellation, and GPU-driven rendering dispatch chains entirely within a single GPU submission.

## F-3.1.9 Concurrent Binary Trees (CBT)

Binary tree-based adaptive tessellation framework for arbitrary polygon meshes. The binary tree structure enables parallel subdivision and simplification decisions on the GPU, achieving planetary-scale geometry rendering from coarse input meshes with sub-millisecond tessellation times.

## F-3.1.10 Automatic LOD Generation

Offline and runtime mesh simplification that generates LOD chains from high-poly source meshes. Uses edge-collapse simplification with quadric error metrics to produce progressively coarser meshlet hierarchies while preserving silhouette edges, UV seams, and material boundaries. Each LOD level is stored as a meshlet group with screen-space error bounds for seamless integration with the GPU LOD selection pipeline.

- **Requirements:** [R-1.1.3](../../requirements/1-architecture/1.1-core-constraints.md) mesh shaders

## F-3.1.11 LOD Blending and Transitions

Smooth transitions between LOD levels to eliminate visual popping. Supports dithered crossfade (stipple pattern blending between outgoing and incoming LOD over a configurable screen-space range), screen-door transparency, and vertex position morphing where vertices interpolate from the coarse LOD position to the fine LOD position based on a blend factor. The transition method is selectable per asset and evaluated in the task/mesh shader pipeline.

- **Requirements:** [R-1.1.3](../../requirements/1-architecture/1.1-core-constraints.md) mesh shaders

## F-3.1.12 Skinned Mesh LOD

Level-of-detail chains for skeletal meshes that reduce both geometric complexity and bone evaluation cost at distance. Each LOD level can specify a reduced bone set, simplified mesh, and lower-fidelity skinning (e.g., fewer bone influences per vertex). LOD selection is screen-size-driven and integrates with the meshlet hierarchy. At extreme distance, skinned meshes can fall back to a rigid impostor or billboard representation.

- **Requirements:** [R-1.1.3](../../requirements/1-architecture/1.1-core-constraints.md) mesh shaders

## F-3.1.13 Static Mesh Merging

Runtime and offline merging of nearby static mesh instances that share the same material into combined meshlet groups. Merged meshes reduce per-object overhead (fewer indirect draw entries, fewer culling tests) while preserving per-meshlet GPU culling granularity. Merge candidates are selected spatially and constrained by material compatibility and maximum merged vertex count. Merging is reversible at runtime when individual instances need to be moved or removed.
