# 2026 Q3 OKRs

Cycle: 2026-07-01 — 2026-09-30. State: Active.

## Context

The 2026-04-12 design review and the 2026-05-20 deslop pass closed every P0 task and most P1
tasks. The remaining work clusters around four themes:

1. Foundational types are owned (canonical-owners) but consumers haven't all migrated.
2. Integration redundancy was reduced but the per-pair docs need spot cleanup.
3. Coverage matrices are seeded but partial; some domains still lack mapping.
4. The backlog is GitHub-ready but unstaffed; issue ownership needs assignment.

These OKRs make those four themes measurable for one quarter. The cycle ends when the retro
section is appended on or about 2026-10-07.

Every shared type lands a single owner with the duplicates removed. Closing the
[canonical-owners](../design/canonical-owners.md) "Pending consolidation" rows is the
deliverable.

## O-2 — Integration redundancy ≤ 2 occurrences per SC rule

pair doc references `shared-conventions.md` and stops restating SC-1..SC-14. The cleanup
isn't done — the compliance line landed but duplicated rule text in many docs has not been
trimmed. This objective measures the trim.

| KR-2.2| Pair docs not linking shared-conventions.md                 |     0    | 0      | done   |
| KR-2.3| Pair docs missing FM-N fallback labels (SC-6)               |    18    | ≤ 4    | open   |
| KR-2.4| Pair docs missing capacity-formula references (SC-7)        |    12    | ≤ 4    | open   |

Backlog: rolled-up under [BL-0033](../backlog/issues/BL-0033-channel-capacity-formula.md).

## O-4 — Backlog is staffed and moving

Open backlog items have owners and observable activity. This objective measures the bridge
from "issue catalog" to "issues in flight". A baseline pull from the 2026-05 audit has 55
open issues; not all need to close this cycle, but each needs an owner and a status
transition.

| ID    | Key Result                                                  | Baseline | Target | Status |
|-------|-------------------------------------------------------------|---------:|--------|--------|
| KR-4.1| Open backlog issues without an owner                        |    55    | ≤ 5    | open   |
| KR-4.2| `status:in-progress` issues at any time during the cycle    |     0    | ≥ 12   | open   |
| KR-4.3| Backlog issues closed during the cycle                      |     0    | ≥ 18   | open   |
| KR-4.4| Issues with `blocked_by` resolved during the cycle          |     0    | ≥ 6    | open   |

Backlog: every `BL-NNNN` in [docs/backlog/issues/](../backlog/index.md).

## Working agreements

- Each KR's `Status` flips to `done` once the target is met. Targets do not retroactively
  shift mid-cycle; if the situation changes, append an Amendment subsection at the bottom of
  this file.
- KRs reference at least one backlog issue, ADR, or design doc. Sandbagging via easy targets
  is anti-pattern.
- Score 0.7 is "good", 1.0 is "lucky", 0.4 or below is "rethink".

## Retro

The retro section is appended on or about 2026-10-07 with:

- Per-objective score (0.0 – 1.0)
- What worked / what didn't
- Carry-overs into 2026-q4
- Net change to the canonical-owner registry, integration trim metrics, coverage breadth,
  and backlog throughput.

This file becomes immutable after the retro is written.
