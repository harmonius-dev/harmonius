1. **US-4.6.1.1** — As a designer (P-5), I want to generate 3D Voronoi fracture patterns at build
   time, so that no runtime mesh generation is needed.
   - **Acceptance:** —
2. **US-4.6.1.2** — As a designer (P-5), I want to seed fracture points using random,
   impact-directed, and artist-guided placement, so that fracture patterns suit the object.
   - **Acceptance:** —
3. **US-4.6.1.3** — As an engine developer (P-26), I want to implement 3D Voronoi decomposition
   producing convex hull fragments, connectivity graph, and joint configs, so that fracture assets
   are complete.
   - **Acceptance:** —
4. **US-4.6.1.4** — As a game developer (P-15), I want entities with a Destructible component to
   reference a fracture asset by handle, so that destruction is data-driven.
   - **Acceptance:** —
5. **US-4.6.1.5** — As a player (P-23), I want destructible objects to shatter into fragments when
   hit, so that destruction feels dynamic.
   - **Acceptance:** —
6. **US-4.6.1.6** — As an engine tester (P-27), I want to confirm mobile caps at 8 fragments per
   fracture asset, so that runtime performance is controlled.
   - **Acceptance:** —
7. **US-4.6.1.7** — As an engine developer (P-26), I want platform-specific LOD variants baked at
   build time, so that fragment count scales with device capability.
   - **Acceptance:** —
8. **US-4.6.1.8** — As a level designer (P-6), I want to place destructible objects with pre-baked
   fracture assets, so that destruction is predictable and performant.
   - **Acceptance:** —
9. **US-4.6.1.9** — As a designer (P-5), I want to place fracture seed points visually in the
   editor, so that art-directed shattering is easy to author.
   - **Acceptance:** —

    - **Acceptance:** —
