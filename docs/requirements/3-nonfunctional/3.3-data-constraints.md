# 3.3 Data Constraints

## R-3.3.1 Reverse-Z Depth

Reverse-Z depth (near=1.0, far=0.0 in NDC) SHALL be used on all platforms. This maximizes floating-point
depth precision at distance, which is critical for large open-world scenes.

## R-3.3.2 Skeleton Bone Limit

The maximum number of bones per skeleton SHALL be 256. This covers all production character rigs while fitting
within a single structured buffer.

## R-3.3.3 Animation Blend Limit

The maximum number of simultaneously blended animation clips per instance SHALL be 4. This is sufficient for
crossfade transitions with additive layers.

## R-3.3.4 IK Chain Length

The maximum IK chain length SHALL be 8 bones, covering standard arm, leg, and spine chains.

## R-3.3.5 Morph Target Precision

Morph target deltas SHALL use half-precision (f16) for position and normal offsets. This provides sub-millimeter
accuracy while halving memory and bandwidth versus f32.

## R-3.3.6 Bindless Heap Capacity

The global bindless descriptor heap SHALL support at least 1 million resource slots to accommodate large
open-world scenes with many unique assets.

## R-3.3.7 Single-Writer Invariant

During the render phase, each resource SHALL have at most one writer per frame. This eliminates the need for
write-write synchronization and simplifies barrier insertion.

## R-3.3.8 Shader IR

A single platform-neutral shader intermediate representation SHALL be used. No hand-authored platform-specific
shader code SHALL exist. All shaders SHALL be cross-compiled from this IR to Metal Shading Language, HLSL,
and SPIR-V.

## R-3.3.9 Shader Graph Format

Shader graphs SHALL be serialized in a human-readable textual format suitable for version control, diffability,
and merge conflict resolution. The format SHALL be deterministic.

## R-3.3.10 Asset Import Format

glTF 2.0 SHALL be the primary 3D asset import format. Standard PBR materials SHALL be automatically converted
to shader graphs.

## R-3.3.11 Lighting Model Selection

Forward+ and deferred lighting SHALL be mutually selectable per render graph. Both SHALL NOT be required to run
simultaneously.

## R-3.3.12 Animation Pipeline Ordering

Morph target evaluation SHALL precede skeletal skinning in the GPU compute pipeline, following standard
glTF/USD conventions.

## R-3.3.13 Constant Buffer Offset Alignment

All constant (uniform) buffer view offsets and sizes SHALL be aligned to 256 bytes. This is the strictest common
requirement across all supported backends: Direct3D 12 mandates 256 bytes
(`D3D12_CONSTANT_BUFFER_DATA_PLACEMENT_ALIGNMENT`), Vulkan guarantees `minUniformBufferOffsetAlignment` does not exceed
256 bytes, and Metal requires 256-byte alignment for practical constant buffer sub-allocation.

## R-3.3.14 Storage Buffer Offset Alignment

All storage (structured/raw) buffer bind offsets SHALL be aligned to 256 bytes. This satisfies the Vulkan
`minStorageBufferOffsetAlignment` upper bound of 256 bytes, the Direct3D 12 raw/structured buffer alignment of 16 bytes
(`D3D12_RAW_UAV_SRV_BYTE_ALIGNMENT`), and Metal's 16-byte storage buffer alignment. The 256-byte value is chosen as the
cross-platform worst case.

## R-3.3.15 Texture Data Placement Alignment

Texture subresource data offsets within staging buffers SHALL be aligned to 512 bytes. This satisfies the Direct3D 12
`D3D12_TEXTURE_DATA_PLACEMENT_ALIGNMENT` of 512 bytes, which is the strictest requirement across all backends.

## R-3.3.16 Texture Row Pitch Alignment

Texture row pitch (bytes per row of texel data) in staging buffers SHALL be aligned to 256 bytes. This satisfies the
Direct3D 12 `D3D12_TEXTURE_DATA_PITCH_ALIGNMENT` of 256 bytes. Vulkan and Metal impose no fixed row pitch alignment but
benefit from aligned rows for DMA throughput.

## R-3.3.17 Heap Resource Placement Alignment

All placed buffer and single-sample texture resources within GPU memory heaps SHALL be aligned to 64 KB (65,536 bytes).
This matches the Direct3D 12 `D3D12_DEFAULT_RESOURCE_PLACEMENT_ALIGNMENT`. Where tighter alignment is available
(Direct3D 12 tight alignment tiers, Metal placement heaps, Vulkan memory type requirements), the backend MAY use a
smaller alignment as an optimization, but the asset pipeline SHALL produce data aligned to the 64 KB worst case.

## R-3.3.18 MSAA Resource Placement Alignment

Placed MSAA texture resources SHALL be aligned to 4 MB (4,194,304 bytes) by default, matching the Direct3D 12
`D3D12_DEFAULT_MSAA_RESOURCE_PLACEMENT_ALIGNMENT`. Where the hardware supports smaller MSAA alignment (64 KB on
Direct3D 12 via `MSAA64KBAlignedTextureSupported`, or runtime-queried values on Vulkan and Metal), the backend MAY use
the smaller alignment.

## R-3.3.19 Asset Chunk Alignment

All asset bundle chunk data SHALL be aligned to 64 KB (65,536 bytes) boundaries. This alignment enables GPU-direct DMA
via DirectStorage, matches sparse texture tile granularity, and ensures page-aligned reads with no copy overhead for
platform-native async IO (IOCP, dispatch_io, io_uring).

## R-3.3.20 Runtime Alignment Query

The GPU backend abstraction SHALL expose functions to query buffer and texture allocation alignment at runtime
(`QueryBufferAllocationInfo`, `QueryTextureAllocationInfo`). Platform-specific alignment values that differ from
the cross-platform worst case SHALL be discoverable at runtime so that backends can use tighter alignment where
supported.

## R-3.3.21 Ray Tracing Acceleration Structure Alignment

Ray tracing acceleration structure resources SHALL be aligned to 256 bytes, matching the Direct3D 12
`D3D12_RAYTRACING_ACCELERATION_STRUCTURE_BYTE_ALIGNMENT`. Shader table start addresses SHALL be aligned to 64 bytes
(`D3D12_RAYTRACING_SHADER_TABLE_BYTE_ALIGNMENT`), and individual shader records SHALL be aligned to 32 bytes
(`D3D12_RAYTRACING_SHADER_RECORD_BYTE_ALIGNMENT`). Vulkan shader group handle alignment SHALL be queried at runtime from
`shaderGroupHandleAlignment` and `shaderGroupBaseAlignment`.
