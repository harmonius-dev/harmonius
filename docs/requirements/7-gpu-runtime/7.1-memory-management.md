# 7.1 Memory Management

## GR-1.1 Unified Memory Allocator

The GPU runtime SHALL provide a unified memory allocator that manages all GPU memory allocation
across all backends. The allocator SHALL be built entirely on top of the GPU backend interface
(`harmonius::gpu`) without depending on third-party memory allocators (VMA, D3D12MA, or
equivalent). The allocator SHALL present a single API regardless of which backend is active.

**Derived from:** R-1.1.5, R-1.1.7

**Rationale:** A first-party allocator eliminates external dependencies, provides consistent
behavior across backends, and enables framework-specific optimization strategies.

## GR-1.2 Block Sub-Allocation

The allocator SHALL support sub-allocating multiple resources from a single GPU heap using a
block allocation strategy. The strategy SHALL achieve O(1) allocation and deallocation with
bounded external fragmentation. Sub-allocations SHALL respect per-resource alignment
requirements queried from the backend.

**Derived from:** R-3.1.8, R-3.4.1

**Rationale:** Creating one heap per resource wastes memory and exceeds GPU heap count limits.
Sub-allocation amortizes heap creation cost and reduces memory waste.

## GR-1.3 Committed Allocation

The allocator SHALL support committed allocations that create a dedicated heap for a single
resource. Committed allocations SHALL be used for resources that require dedicated memory
(e.g., swapchain images, large render targets) or resources that cannot share heaps due to
backend constraints.

**Derived from:** R-3.4.1

**Rationale:** Some resources must have dedicated allocations for correctness or performance.

## GR-1.4 Placed Allocation for Aliasing

The allocator SHALL support placed allocations within shared heaps where multiple resources
bind to the same heap at specified offsets. Placed allocations SHALL be the mechanism for
render graph transient resource aliasing (RG-8.1–8.6).

**Derived from:** RG-8.1, RG-8.2, RG-8.5

**Rationale:** Transient render targets with non-overlapping lifetimes can share the same heap
range, reducing peak VRAM usage.

## GR-1.5 Ring Buffer Allocation

The allocator SHALL provide ring buffer allocation for per-frame staging and constant buffers.
Ring allocation SHALL partition a pre-allocated buffer into per-frame slots and recycle each
slot after the corresponding frame's GPU work completes, as determined by timeline fence
queries. Ring allocation SHALL perform zero heap allocations on the per-frame hot path.

**Derived from:** R-3.1.8, RG-8.4, RG-10.5

**Rationale:** Per-frame constant and staging data requires fast, contention-free allocation
with automatic recycling.

## GR-1.6 Online Defragmentation

The allocator SHALL provide online defragmentation that compacts heap usage by relocating
resources to eliminate fragmentation gaps. Defragmentation SHALL be incremental (processing a
bounded number of moves per frame) and SHALL NOT stall the GPU pipeline. Resource relocations
SHALL be performed via GPU copy commands coordinated with the consumer's fence timeline.

**Derived from:** R-3.1.8, R-3.4.1

**Rationale:** Long-running applications with dynamic resource creation and destruction
accumulate fragmentation that must be resolved without frame hitches.

## GR-1.7 Memory Budget Tracking

The allocator SHALL query per-heap-type memory budgets from the backend and track current usage
against those budgets. The allocator SHALL expose budget usage metrics (total budget, current
usage, peak usage) for diagnostic reporting. When an allocation would exceed the budget, the
allocator SHALL return an error rather than silently over-committing.

**Derived from:** R-3.4.1, RG-12.6

**Rationale:** GPU memory is finite and shared with other processes; over-commitment causes
performance degradation or crashes.

## GR-1.8 Heap Type Selection

The allocator SHALL automatically select the appropriate memory heap type (device-local,
host-visible, host-cached) based on the resource's declared usage flags. Device-local heaps
SHALL be used for GPU-only resources, host-visible heaps for staging and upload buffers, and
host-cached heaps for readback buffers. Sub-allocation SHALL only occur between resources
assigned to the same heap type.

**Derived from:** RG-8.5

**Rationale:** Different memory types serve different access patterns; mixing them in the same
heap is invalid or inefficient.

## GR-1.9 Allocation Metadata Query

The allocator SHALL support querying metadata for any active allocation, including: allocated
size, required alignment, heap type, offset within heap, and heap handle. This metadata SHALL
be available without additional GPU queries after the initial allocation.

**Derived from:** RG-8.6, RG-12.6

**Rationale:** Diagnostics, defragmentation, and aliasing decisions require allocation metadata.

## GR-1.10 Pool-Scoped Allocation

The allocator SHALL support creating named allocation pools with configurable maximum capacity
and element size class. Allocations from a pool SHALL be served from the pool's reserved heap
capacity. Pool exhaustion SHALL be reported to the caller as an error, enabling the caller to
invoke eviction callbacks before retrying.

**Derived from:** RG-2.8, RG-11.5

**Rationale:** Streaming resource pools (tiles, voxels, chunks) require bounded memory usage
with deterministic allocation behavior.

## GR-1.11 Sparse Resource Binding

The allocator SHALL support managing sparse resource tile bindings, mapping virtual texture
tiles to physical heap pages. The allocator SHALL track which tiles are currently bound and
provide batch bind/unbind operations that map to the backend's sparse binding API.

**Derived from:** RG-2.9

**Rationale:** Virtual textures and partially-resident resources require fine-grained tile
binding management.
