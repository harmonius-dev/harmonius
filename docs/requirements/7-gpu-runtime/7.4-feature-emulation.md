# 7.4 Feature Emulation

## GR-4.1 Capability-Aware Command Recording

The GPU runtime SHALL provide a capability-aware command recorder that checks device
capabilities and automatically selects native or emulated code paths for each operation.
Callers SHALL use a single API regardless of which features the current backend supports.

**Derived from:** R-1.1.5, R-1.1.7

**Rationale:** Higher-level systems should not contain per-backend conditional logic; feature
adaptation is the GPU runtime's responsibility.

## GR-4.2 Split Barrier Emulation

When `DeviceCapabilities::split_barriers` is false, the GPU runtime SHALL emulate split
barriers by deferring the begin-barrier and emitting a single immediate barrier at the
split-end site with the combined source and destination stages. When split barriers are
supported, the GPU runtime SHALL forward begin and end calls directly to the backend.

**Derived from:** RG-3.3

**Rationale:** Split barriers improve GPU utilization on D3D12 and Vulkan but are unsupported
on Metal. The GPU runtime provides a uniform API.

## GR-4.3 Barrier Batching

The GPU runtime SHALL batch multiple barriers targeting the same synchronization scope into a
single backend API call. Consecutive barrier requests between the same pair of pipeline stages
SHALL be merged into one `barrier()` call with a combined resource list.

**Derived from:** R-3.1.2, RG-3.6

**Rationale:** Individual barrier calls have per-call overhead; batching reduces CPU cost
and enables backend-side optimization.

## GR-4.4 Barrier Deduplication

The GPU runtime SHALL deduplicate barrier requests by consulting the resource state cache
(GR-2.6). If a resource is already in the target state, the barrier SHALL be elided entirely.
If a barrier is a subset of a previously emitted barrier within the same command buffer, it
SHALL be suppressed.

**Derived from:** R-3.1.2, RG-3.6

**Rationale:** The render graph compiler emits barriers conservatively; runtime deduplication
removes unnecessary transitions without complicating the compiler.

## GR-4.5 Queue Ownership Transfer Elision

When the backend reports unified memory architecture (e.g., Metal), the GPU runtime SHALL
elide queue ownership transfer barriers, as cross-queue resource sharing does not require
explicit ownership handoff on unified memory systems.

**Derived from:** RG-3.4

**Rationale:** Metal's unified memory does not require queue ownership transfers; emitting
them is unnecessary overhead.
