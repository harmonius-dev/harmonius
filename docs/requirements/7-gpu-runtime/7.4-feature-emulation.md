# 7.4 Feature Emulation

## GR-4.1 Capability-Aware Command Recording

The GPU runtime SHALL provide a capability-aware command recorder that checks device
capabilities and automatically selects native or emulated code paths for each operation.
Callers SHALL use a single API regardless of which features the current backend supports.

**Derived from:** R-1.1.5, R-1.1.7

**Rationale:** Higher-level systems should not contain per-backend conditional logic; feature
adaptation is the GPU runtime's responsibility.

## GR-4.2 Split Barrier Emulation

When `DeviceCapabilities::split_barriers` is false, the GPU runtime SHALL emulate split
barriers by deferring the begin-barrier and emitting a single immediate barrier at the
split-end site with the combined source and destination stages. When split barriers are
supported, the GPU runtime SHALL forward begin and end calls directly to the backend.

**Derived from:** RG-3.3

**Rationale:** Split barriers improve GPU utilization on D3D12 and Vulkan but are unsupported
on Metal. The GPU runtime provides a uniform API.

## GR-4.3 Barrier Batching

The GPU runtime SHALL batch multiple barriers targeting the same synchronization scope into a
single backend API call. Consecutive barrier requests between the same pair of pipeline stages
SHALL be merged into one `Barrier()` call with a combined resource list.

**Derived from:** R-3.1.2, RG-3.6

**Rationale:** Individual barrier calls have per-call overhead; batching reduces CPU cost
and enables backend-side optimization.

## GR-4.4 Barrier Deduplication

The GPU runtime SHALL deduplicate barrier requests by consulting the resource state cache
(GR-2.6). If a resource is already in the target state, the barrier SHALL be elided entirely.
If a barrier is a subset of a previously emitted barrier within the same command buffer, it
SHALL be suppressed.

**Derived from:** R-3.1.2, RG-3.6

**Rationale:** The render graph compiler emits barriers conservatively; runtime deduplication
removes unnecessary transitions without complicating the compiler.

## GR-4.5 Queue Ownership Transfer Elision

When the backend reports unified memory architecture (e.g., Metal), the GPU runtime SHALL
elide queue ownership transfer barriers, as cross-queue resource sharing does not require
explicit ownership handoff on unified memory systems.

**Derived from:** RG-3.4

**Rationale:** Metal's unified memory does not require queue ownership transfers; emitting
them is unnecessary overhead.

## GR-4.6 Ray Tracing Pipeline Dispatch Emulation

When `DeviceCapabilities::ray_tracing` is false and `DeviceCapabilities::ray_tracing_inline`
is true, the GPU runtime SHALL emulate `TraceRays()` dispatches by translating them into
compute shader `Dispatch()` calls that use inline ray queries. The emulation SHALL accept the
same `TraceRaysDesc` parameters as the native path and compute equivalent thread group
dimensions for the compute dispatch. The translation SHALL be transparent to the render graph
execution engine.

**Derived from:** R-1.1.5, R-1.1.7, R-2.5.1

**Rationale:** Metal does not support dedicated ray tracing pipelines but supports inline ray
queries in compute shaders. The GPU runtime must bridge this gap so the render graph can
express RT passes uniformly across all backends.

## GR-4.7 Shader Binding Table Emulation

When emulating ray tracing pipelines via compute shaders, the GPU runtime SHALL replace the
native shader binding table (SBT) with a flat material data buffer accessible via bindless
resource indexing. Each SBT hit group record's per-record data SHALL be packed into a GPU
buffer indexed by geometry instance ID and geometry index. Miss shader and callable shader
data SHALL be stored in separate buffer regions. The emulated SBT buffer SHALL be rebuilt only
when the SBT layout changes, not per frame.

**Derived from:** GR-4.6, R-2.5.2

**Rationale:** The SBT is a D3D12/Vulkan-specific mechanism for binding shaders and data to
ray-geometry intersections. On backends without dedicated RT pipelines, equivalent data must
be accessible through the standard bindless resource model.

## GR-4.8 RT Pipeline Pair Registration

The GPU runtime SHALL support registering pipeline pairs consisting of a native ray tracing
pipeline handle and a corresponding compute shader fallback pipeline handle. At dispatch time,
the GPU runtime SHALL select the appropriate pipeline based on
`DeviceCapabilities::ray_tracing`. Pipeline pairs SHALL be cached and associated by a
caller-provided identifier. Callers SHALL register both pipeline variants at pipeline creation
time; the GPU runtime SHALL NOT perform shader compilation or translation.

**Derived from:** GR-4.1, GR-4.6, R-1.1.7

**Rationale:** The GPU runtime operates on opaque pipeline handles and does not perform shader
compilation. Both native and fallback pipelines must be provided externally (by the shader
pipeline module) and registered so the runtime can select transparently.

## GR-4.9 Acceleration Structure Binding Adaptation

When emulating ray tracing pipelines via compute shaders, the GPU runtime SHALL bind
acceleration structures as shader resource views accessible to compute shaders for inline ray
queries. On native ray tracing backends, acceleration structures SHALL be bound via the ray
tracing pipeline's descriptor layout. The binding method selection SHALL be based on
`DeviceCapabilities::ray_tracing` and SHALL be transparent to the caller.

**Derived from:** GR-4.6, R-2.5.2

**Rationale:** Inline ray queries access acceleration structures as regular shader resources
rather than through the dedicated RT pipeline binding model. The GPU runtime must adapt the
binding approach based on the dispatch path.
