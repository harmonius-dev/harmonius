# 7.3 Work Graph Runtime

## GR-3.1 Transparent Work Graph Execution

The GPU runtime SHALL provide a work graph executor that can drive render graph pass execution
using either native GPU work graphs or CPU-side emulation. The choice between native and
emulated paths SHALL be made automatically based on `DeviceCapabilities::work_graphs` and
SHALL be transparent to the render graph execution engine.

**Derived from:** R-2.7.13, RG-1.13

**Rationale:** GPU work graphs enable the GPU to self-schedule pass execution, eliminating
CPU-side command buffer recording overhead. When unavailable, the CPU must perform equivalent
scheduling.

## GR-3.2 Native Work Graph Path

When `DeviceCapabilities::work_graphs` is true, the work graph executor SHALL translate the
render graph's compiled execution plan into a GPU work graph program. Each render graph pass
SHALL be mapped to a work graph node, with inter-pass dependencies expressed as work graph
edges. The GPU SHALL self-schedule execution of all nodes within the work graph without CPU
involvement after initial dispatch.

**Derived from:** R-2.7.13

**Rationale:** Native GPU work graphs eliminate CPU command recording latency and enable the
GPU to dynamically schedule and load-balance work.

## GR-3.3 CPU-Side Emulation Path

When `DeviceCapabilities::work_graphs` is false, the work graph executor SHALL provide
CPU-side emulation that records command buffers per pass and submits them in the order
determined by the render graph's topological sort. The emulation path SHALL produce
identical rendering results to the native path.

**Derived from:** R-2.7.13

**Rationale:** Vulkan and Metal backends do not support GPU work graphs; the CPU must perform
equivalent scheduling on these platforms.

## GR-3.4 Unified Executor API

The work graph executor SHALL present a single execution API that accepts a compiled execution
plan and device capabilities. Callers SHALL NOT need to check capabilities or select paths
manually. The executor SHALL return the same result types regardless of which path is used.

**Derived from:** R-1.1.7, R-5.1.2

**Rationale:** The render graph execution engine should not contain work-graph-specific logic;
path selection is the GPU runtime's responsibility.

## GR-3.5 Incremental Plan Translation

The work graph executor SHALL cache the translated GPU work graph program and re-translate
only when the execution plan changes (i.e., after graph recompilation). Per-frame execution
SHALL reuse the cached program without re-translation.

**Derived from:** R-3.1.8

**Rationale:** Work graph program creation involves shader compilation and memory allocation;
it must be amortized over many frames, not repeated every frame.

## GR-3.6 Per-Frame Data Injection

The work graph executor SHALL support injecting per-frame data (resource bindings, constants,
activation flags) into the work graph program before dispatch. On the native path, per-frame
data SHALL be passed via the work graph's root input record. On the emulated path, per-frame
data SHALL be bound to each pass's command buffer as push constants or descriptor updates.

**Derived from:** RG-14.1, RG-14.2, RG-14.3

**Rationale:** Per-frame data changes every frame; the work graph executor must accept fresh
data without re-creating the program.

## GR-3.7 Diagnostic Instrumentation

The work graph executor SHALL support diagnostic timestamp insertion on both paths. On the
native path, timestamps SHALL be inserted at the work graph dispatch boundaries. On the
emulated path, timestamps SHALL be inserted per pass, matching the render graph's diagnostics
requirements (RG-12.1).

**Derived from:** RG-12.1, RG-12.7

**Rationale:** Performance profiling must work identically regardless of execution path.

## GR-3.8 Synchronization Fidelity

The emulated work graph path SHALL insert explicit pipeline barriers and timeline fence signals
that replicate the implicit synchronization guarantees of native GPU work graphs.
Producer-consumer dependencies in the execution plan SHALL be enforced via pipeline barriers
between consecutive passes on the same queue. Cross-queue dependencies SHALL be enforced via
timeline fence signal-and-wait pairs. The barrier and fence placement SHALL be derived from
the execution plan's dependency edges and resource access declarations.

**Derived from:** RG-3.1, RG-3.3, GR-3.3

**Rationale:** Native GPU work graphs provide implicit synchronization between dependent nodes.
The emulated path must guarantee identical ordering and visibility semantics using explicit
synchronization primitives.

## GR-3.9 Backing Memory Management

When using the native work graph path, the work graph executor SHALL allocate backing memory
for intermediate node records from `gpu_runtime::memory::Allocator`. The backing memory size
SHALL be determined by querying the GPU work graph program's memory requirements via the
backend. Backing memory SHALL be allocated once per execution plan and reused across frames.
Reallocation SHALL occur only when the execution plan changes and the new program requires
more backing memory than currently allocated.

**Derived from:** GR-1.1, GR-3.2, GR-3.5

**Rationale:** GPU work graph programs require scratch memory for node record storage and
scheduling metadata. This memory must be managed through the unified allocator to participate
in budget tracking and defragmentation.
