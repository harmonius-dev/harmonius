# 3.1 Testing and CI

## 3.1.1 Real GPU Required

All GPU-exercising tests SHALL require real GPU hardware. No software rasterizer fallback SHALL exist. Tests SHALL explicitly skip (not silently pass) when no suitable GPU is found.

## 3.1.2 Test-Driven Development

All modules SHALL follow test-first development: write a failing test, implement minimal code to pass, refactor. No production code SHALL be written without a motivating test.

## 3.1.3 Snapshot Validation

End-to-end rendering tests SHALL capture GPU output via readback and compare against golden reference images using per-pixel diff with a statistical error metric (p99). Per-platform golden baselines SHALL be maintained independently.

## 3.1.4 Fuzz Testing

Serialization, graph construction, and asset import paths SHALL be fuzz-tested to ensure no panics or undefined behavior on arbitrary input. All malformed input SHALL produce typed errors.

## 3.1.5 GPU CI on All Platforms

CI SHALL run on GPU hardware for each target platform (macOS/Metal, Linux/Vulkan, Windows/D3D12). Per-commit gates SHALL include unit tests, E2E GPU tests, snapshot regression, and memory leak detection.

## 3.1.6 Performance Regression Detection

GPU frame time SHALL be tracked per commit. A statistically significant increase exceeding 5% over a rolling baseline SHALL be flagged as a regression.

## 3.1.7 Instrumentation Zero-Overhead

All profiling and debugging instrumentation SHALL be opt-in at compile time. When disabled, instrumentation SHALL have zero runtime overhead in release builds.

## 3.1.8 Validation Layer Enforcement

GPU validation layer errors SHALL be treated as test failures on all platforms. No validation warnings SHALL be suppressed in CI.

## 3.1.9 Stress Testing

The system SHALL be stress-tested for: maximum resource allocation and deallocation, maximum concurrent streaming requests, maximum draw count via indirect dispatch, and near-full descriptor heap capacity.
