# R-8.NFR -- Networking Non-Functional Requirements
Non-functional requirements for the networking domain covering latency budgets, throughput targets,
scalability limits, bandwidth constraints, and reliability thresholds. These complement the
functional requirements in R-8.1 through R-8.8 and trace to the cross-cutting performance budget in
R-X.1.1.

## Latency

| ID | Requirement | Derived From | Rationale | Verification |
|----|-------------|--------------|-----------|--------------|
| R-8.NFR.1 | R-8.NFR.1 End-to-End Input Latency: Client-side prediction (R-8.4.1) **SHALL** make player inputs appear to execute within 1 render frame (< 16.7 ms at 60 fps) regardless of network round-trip time. Server reconciliation and input replay **SHALL** complete within a single simulation tick with no visible teleport or stutter for corrections under 50 ms of positional drift. | F-8.4.1 (Input Prediction), R-X.1.1 (Frame Time Budget) | Perceived input latency is the single most important factor in action gameplay feel. Any perceptible delay breaks the connection between input and response. | Integration test: simulate 150 ms RTT. Verify local input response is within 1 frame. Inject a 30 ms positional correction and verify no visible snap. Inject a 100 ms correction and verify smooth convergence within 200 ms. |
| R-8.NFR.2 | R-8.NFR.2 Zone Transition Latency: Seamless zone transitions (R-8.7.2) **SHALL** complete the state handoff and connection rebind within 500 ms, with no loading screen, no disconnect, and no visible state discontinuity. Player migration between servers (R-8.7.4) **SHALL** complete within 100 ms. | F-8.7.2 (Seamless Zone Transitions), F-8.7.4 (Player Migration) | Zone transitions that cause visible pauses or disconnects break open-world immersion and are unacceptable in a seamless MMO. | Integration test: move a player across a zone boundary. Measure total handoff time and verify it is under 500 ms. Verify the client renders continuously during the transition. Measure migration handoff and verify it is under 100 ms. |
| R-8.NFR.3 | R-8.NFR.3 Reconnection Latency: Session reconnection (R-8.5.5) **SHALL** restore the player to their exact prior state within 3 seconds of network restoration, provided the disconnect duration is within the configured grace window (default 120 seconds). | F-8.5.5 (Session Reconnection) | Slow reconnection frustrates players returning after transient network issues and may cause them to abandon the session. | Integration test: simulate a 10-second disconnect. Measure time from network restoration to full state restoration and verify it is under 3 seconds. Verify position, buffs, group membership, and combat state are all restored. |

## Throughput

| ID | Requirement | Derived From | Rationale | Verification |
|----|-------------|--------------|-----------|--------------|
| R-8.NFR.4 | R-8.NFR.4 Replication Throughput: The state replication system **SHALL** replicate at least 100,000 entity updates per second per server process, with delta compression (R-8.2.1) achieving at least 80% bandwidth reduction compared to full-state replication. | F-8.2.1 (Delta Replication), F-8.2.5 (Priority Scheduling) | MMO zones with thousands of entities and hundreds of clients require high replication throughput to maintain world state consistency. | Benchmark: replicate 10,000 entities to 100 clients with 5% per-tick change rate. Measure updates per second and verify at least 100,000. Measure bandwidth and verify at least 80% reduction vs. full-state baseline. |
| R-8.NFR.5 | R-8.NFR.5 RPC Throughput: The RPC system **SHALL** process at least 50,000 server RPCs per second per server process without exceeding the networking frame budget defined by R-X.1.1 (default 0.5 ms at 60 fps for networking). | F-8.3.1 (Server RPC), F-8.3.5 (Parameter Validation) | Large-scale PvP encounters generate bursts of ability activations and interactions that must be processed within the frame budget. | Benchmark: submit 50,000 RPCs per second with 5-parameter payloads. Verify all are validated and executed. Verify networking processing time stays within the 0.5 ms budget. |
| R-8.NFR.6 | R-8.NFR.6 Database Write Throughput: The persistence layer (R-8.7.5) **SHALL** sustain at least 10,000 transactional writes per second across the server fleet without blocking the game simulation tick. Write latency **SHALL** be below 10 ms at the 99th percentile. | F-8.7.5 (Persistent World State) | Character saves, inventory updates, auction transactions, and quest progress writes must complete at scale without impacting gameplay. | Benchmark: execute 10,000 concurrent character save transactions per second. Verify no simulation tick is blocked. Measure p99 write latency and verify it is below 10 ms. |

## Scalability

| ID | Requirement | Derived From | Rationale | Verification |
|----|-------------|--------------|-----------|--------------|
| R-8.NFR.7 | R-8.NFR.7 Concurrent Connections per Server Process: Each server process **SHALL** support at least 10,000 simultaneous authenticated connections with O(1) per-connection overhead for keepalive, timeout detection, and heartbeat processing, per R-8.1.2. | F-8.1.2 (Connection Lifecycle) | MMO server density depends on per-connection overhead remaining constant. A server supporting fewer connections requires more hardware for the same population. | Load test: establish 10,000 connections. Verify CPU cost per connection for keepalive processing does not increase with connection count. Verify total keepalive processing time is below 0.1 ms per tick. |
| R-8.NFR.8 | R-8.NFR.8 Matchmaking Queue Scalability: The matchmaking system (R-8.5.2) **SHALL** handle at least 50,000 concurrent queue entries across all regions and match types, completing match formation within 120 seconds for 95% of queue entries. | F-8.5.2 (Matchmaking) | Peak-hour matchmaking queues during game launches and content releases can reach tens of thousands of entries. Long queue times cause player attrition. | Load test: enqueue 50,000 players. Verify 95% are matched within 120 seconds. Verify matchmaker CPU does not exceed its allocated budget. |
| R-8.NFR.9 | R-8.NFR.9 Server Mesh Scalability: The dynamic server mesh (R-8.7.3) **SHALL** support at least 100 server processes per world shard, dynamically splitting and merging regions based on load, with split operations completing within 10 seconds and merge operations within 30 seconds. | F-8.7.3 (Dynamic Server Mesh) | Large-scale events (guild sieges, world bosses) concentrate players in a small area. The server mesh must split fast enough to prevent overload. | Integration test: simulate a 500-player convergence. Verify region split triggers within 5 seconds of overload detection and completes within 10 seconds. Disperse players and verify merge completes within 30 seconds. |

## Bandwidth

| ID | Requirement | Derived From | Rationale | Verification |
|----|-------------|--------------|-----------|--------------|
| R-8.NFR.10 | R-8.NFR.10 Per-Client Bandwidth Budget: The replication system **SHALL** enforce a configurable per-client bandwidth budget (default 128 KBps downstream, 32 KBps upstream) that is never exceeded, with priority scheduling (R-8.2.5) ensuring the most gameplay-critical entities consume available bandwidth first. | F-8.2.5 (Priority Scheduling), F-8.1.7 (Congestion Control) | Many players are on consumer internet connections with limited bandwidth. Exceeding per-client limits causes packet loss and degraded gameplay. | Integration test: set a 128 KBps budget with 500 entities needing replication. Verify downstream bandwidth never exceeds 128 KBps. Verify high-priority entities receive updates every tick while low-priority entities are deferred. |
| R-8.NFR.11 | R-8.NFR.11 Encryption Throughput: DTLS 1.3 encryption (R-8.1.5) **SHALL** achieve at least 1 Gbps throughput with hardware- accelerated AES-GCM on supported platforms, and at least 100 Mbps in software fallback, with encryption adding no more than 5% CPU overhead relative to unencrypted packet processing. | F-8.1.5 (DTLS Encryption) | Encryption is mandatory for all traffic. Insufficient throughput would bottleneck inter-server communication on datacenter links. | Benchmark: measure encryption throughput on hardware with AES-NI and verify at least 1 Gbps. Measure on software fallback and verify at least 100 Mbps. Measure CPU overhead and verify below 5%. |

## Reliability

| ID | Requirement | Derived From | Rationale | Verification |
|----|-------------|--------------|-----------|--------------|
| R-8.NFR.12 | R-8.NFR.12 Reliable Channel Delivery Guarantee: The reliable ordered channel (R-8.1.3) **SHALL** deliver 100% of messages in order with zero loss under network conditions of up to 15% packet loss and 200 ms RTT. Retransmission latency **SHALL** be within 1.5x of the measured RTT. | F-8.1.3 (Reliable Ordered Channel) | Critical gameplay events (trades, quest completions, ability confirmations) must be delivered reliably even under poor network conditions. | Integration test: send 100,000 numbered messages over a link with 15% loss and 200 ms RTT. Verify 100% delivery in order. Measure retransmission latency and verify it is within 1.5x RTT. |
| R-8.NFR.13 | R-8.NFR.13 Server Availability: The server cluster (R-8.5.4) **SHALL** maintain 99.9% availability (< 8.77 hours downtime per year), with automatic failover replacing crashed server processes within 30 seconds and rolling restarts completing without disconnecting any player. | F-8.5.4 (Cluster Management), F-8.7.6 (Auto-Scaling) | MMO players expect near-continuous availability. Extended downtime causes player attrition and revenue loss. | Chaos test: kill a server process during peak load. Verify replacement is provisioned and accepting players within 30 seconds. Verify affected players are migrated without disconnection. Initiate a rolling restart and verify zero player disconnections. |
| R-8.NFR.14 | R-8.NFR.14 Anti-Cheat False Positive Rate: The server-side cheat detection system (R-8.8.1) **SHALL** maintain a false positive rate below 0.1% (no more than 1 false flag per 1,000 legitimate players) when detection thresholds account for legitimate network latency up to 200 ms and prediction correction drift up to 50 ms of positional error. | F-8.8.1 (Server-Side Cheat Detection) | False positives erode player trust and generate support burden. Detection thresholds must accommodate legitimate network conditions. | Load test: simulate 10,000 legitimate clients with RTTs ranging from 20 ms to 200 ms. Verify no more than 10 are falsely flagged over a 60-minute test. Simulate 100 cheating clients and verify at least 95% are correctly detected. |
