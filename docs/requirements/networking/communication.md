# R-8.9 — Communication Framework Requirements

## Channel System

| ID | Requirement | Derived From | Rationale | Verification |
|----|-------------|--------------|-----------|--------------|
| R-8.9.1 | Communication Channel System: The engine **SHALL** provide a unified channel abstraction shared by game runtime and editor that supports text, voice, or both simultaneously per channel. Each channel **SHALL** have a unique ID, type (global, team, party, whisper, custom, editor), membership list, and permission set. Channels **SHALL** persist across zone transitions (F-8.7.2) and server migrations (F-8.7.4). The same channel API **SHALL** function identically in game runtime and editor contexts. |  [F-8.9.1](../../features/networking/communication.md) | A single channel abstraction eliminates duplicate communication infrastructure across game and editor, reducing maintenance burden and ensuring consistent behavior. | Integration test: create channels of each type (global, team, party, whisper, custom, editor) and verify membership operations (join, leave). Trigger a zone transition and verify channel membership survives. Verify the same API works from both game runtime and editor client contexts. |
| R-8.9.1a | Channel Capacity: The engine **SHALL** support at least 256 concurrent channels per server process and at least 64 simultaneous channel memberships per client. |  [F-8.9.1](../../features/networking/communication.md) | MMO social structures (guilds, parties, raids, custom channels, whispers) require many concurrent channels. | Load test: create 256 channels with 100 members each and verify no channel operations degrade below performance targets. Verify a single client can join 64 channels and receive messages on all of them. |

## Text Chat

| ID | Requirement | Derived From | Rationale | Verification |
|----|-------------|--------------|-----------|--------------|
| R-8.9.2 | Text Chat: The engine **SHALL** deliver text messages over communication channels via the reliable ordered channel (F-8.1.3) with server-side persistence and full-text search. Messages **SHALL** support @-mentions, inline asset references, and emoji. Chat history **SHALL** load on channel join. |  [F-8.9.2](../../features/networking/communication.md) | Text chat is the primary communication method for players who cannot or prefer not to use voice, and provides a persistent record of conversations. | Integration test: send a text message on each channel type and verify delivery to all members. Verify @-mentions trigger notifications on the mentioned user's client. Verify chat history loads on channel join with correct ordering. Search chat history for a keyword and verify correct results. |
| R-8.9.2a | Text Chat Delivery Latency: The engine **SHALL** deliver text messages to all channel members within 100 ms under normal network conditions (< 50 ms RTT, < 1% packet loss). |  [F-8.9.2](../../features/networking/communication.md) | Text chat must feel instantaneous to maintain conversational flow; delays above 100 ms are perceptible. | Benchmark: send 1,000 messages on a channel with 50 members and measure delivery latency. Assert p99 latency is below 100 ms on a link with 20 ms RTT. |
| R-8.9.2b | Chat History Persistence: The engine **SHALL** persist chat history server-side with a configurable retention period (default 30 days). History **SHALL** be queryable with full-text search returning results within 200 ms for indices up to 10 million messages. |  [F-8.9.2](../../features/networking/communication.md) | Persistent chat history enables players to review past conversations and provides evidence for moderation. | Integration test: insert 10 million messages into the chat store. Execute a full-text search and verify results return within 200 ms. Verify messages older than the retention period are pruned. |

## Voice Chat

| ID | Requirement | Derived From | Rationale | Verification |
|----|-------------|--------------|-----------|--------------|
| R-8.9.3 | Voice Chat Integration: The engine **SHALL** integrate voice channels into the unified channel system, transmitting Opus-encoded audio (F-5.5.1) over unreliable unordered channels (F-8.1.4), decoding into the mixer bus hierarchy (F-5.1.3). Proximity channels **SHALL** spatialize audio using the shared BVH (F-1.9.8). Party and team channels **SHALL** mix as non-spatialized bus sends. VAD (F-5.5.3) **SHALL** gate transmission. AEC (F-5.5.9) **SHALL** prevent feedback. |  [F-8.9.3](../../features/networking/communication.md) | Voice chat shares the same channel abstraction as text, enabling unified membership, permissions, and moderation across both modalities. | Integration test: join a voice channel and transmit audio. Verify decoded audio appears in the mixer at the correct bus. Verify proximity channels attenuate with distance. Verify VAD gates transmission during silence. Verify per-speaker volume and mute controls function. |
| R-8.9.3a | Voice Latency: The engine **SHALL** deliver voice audio with mouth-to-ear latency not exceeding 150 ms under normal network conditions (< 50 ms RTT, < 1% packet loss), inclusive of capture, encoding, network transit, jitter buffer, decoding, and mixer output. |  [F-8.9.3](../../features/networking/communication.md) | Voice latency above 150 ms causes conversational overlap and frustration; below 150 ms feels natural. | Benchmark: measure mouth-to-ear latency on a loopback with 50 ms simulated RTT. Assert total latency is below 150 ms. |
| R-8.9.3b | Voice Packet Delivery: Voice packets **SHALL** be delivered with one-way latency not exceeding 50 ms exclusive of jitter buffer depth, using unreliable unordered delivery (F-8.1.4). | [F-8.9.3](../../features/networking/communication.md), [F-8.9.7](../../features/networking/communication.md) | Network-layer voice packet latency must be minimal to leave headroom for encoding, jitter buffer, and decoding within the 150 ms end-to-end budget. | Integration test: send voice packets on a link with 20 ms RTT and measure one-way delivery time. Assert p99 is below 50 ms. |

## Direct Messaging

| ID | Requirement | Derived From | Rationale | Verification |
|----|-------------|--------------|-----------|--------------|
| R-8.9.4 | Direct Messaging: The engine **SHALL** support private 1:1 conversations as special-case channels with exactly two members. Direct messages **SHALL** be end-to-end encrypted using keys derived from participants' session keys. DM history **SHALL** persist server-side (encrypted at rest) and be accessible across sessions. Offline messages **SHALL** be queued and delivered on next login. |  [F-8.9.4](../../features/networking/communication.md) | Players need private communication that is secure, persistent, and works across sessions and login states. | Integration test: start a DM between two users and verify message delivery. Verify messages are end-to-end encrypted by inspecting wire-level payloads. Send a message to an offline user, log them in, and verify delivery. Verify DM history loads on reconnection. Escalate a text DM to a voice call and verify audio flows. |
| R-8.9.4a | DM Encryption: DM messages **SHALL** use end-to-end encryption with a key derived from the participants' session keys using X25519 Diffie-Hellman key exchange. The server **SHALL NOT** have access to plaintext message content. Messages at rest **SHALL** be encrypted with AES-256-GCM. |  [F-8.9.4](../../features/networking/communication.md) | Private messages must be confidential even from the server operator to protect player privacy. | Integration test: capture DM traffic at the server and verify no plaintext content is extractable. Verify stored messages are encrypted at rest. Verify key derivation uses X25519. |

## Chat Moderation

| ID | Requirement | Derived From | Rationale | Verification |
|----|-------------|--------------|-----------|--------------|
| R-8.9.5 | Chat Moderation: The engine **SHALL** provide server-side moderation for text and voice channels: configurable profanity filter with locale-aware word lists, per-user mute (text and voice independently), block (prevents all communication from blocked user), report (flags user with evidence for admin review), and rate limiting (messages per second per user per channel). All moderation actions **SHALL** be logged with timestamps for audit. |  [F-8.9.5](../../features/networking/communication.md) | Moderation tools protect the community from harassment, spam, and toxic behavior while providing admins with auditable evidence for sanctions. | Integration test: configure a profanity filter and send a message containing a filtered word; verify the word is replaced or the message is blocked. Mute a user and verify they cannot send text or voice in the channel. Block a user and verify all their messages are hidden from the blocker. Submit a report and verify it appears in the admin moderation queue with evidence. Send messages exceeding the rate limit and verify throttling. |
| R-8.9.5a | Profanity Filter Performance: The profanity filter **SHALL** evaluate a message against the full word list in under 1 ms per message, supporting word lists of up to 50,000 entries across all configured locales. |  [F-8.9.5](../../features/networking/communication.md) | The filter runs on every outbound message and must not add perceptible latency to chat delivery. | Benchmark: load a 50,000-entry word list and filter 10,000 messages. Assert p99 filter time is below 1 ms per message. |
| R-8.9.5b | Moderation Rate Limiting: The engine **SHALL** enforce configurable per-user per-channel message rate limits with a default of 5 messages per second and a burst allowance of 10 messages. Exceeding limits **SHALL** trigger throttling; sustained abuse **SHALL** escalate to temporary mute. |  [F-8.9.5](../../features/networking/communication.md) | Rate limiting prevents chat spam without penalizing normal conversation patterns. | Integration test: send 6 messages in 1 second and verify the 6th is throttled. Send 11 messages in a burst and verify the 11th is throttled. Sustain abuse for 30 seconds and verify automatic temporary mute. |

## VR Chat Integration

| ID | Requirement | Derived From | Rationale | Verification |
|----|-------------|--------------|-----------|--------------|
| R-8.9.6 | VR Chat Integration: The engine **SHALL** position voice audio sources at avatar head positions in 3D space using HRTF binaural rendering (F-5.2.3) for VR contexts. Text input **SHALL** use a virtual keyboard rendered in VR worldspace with controller or hand-tracking input. Chat bubbles **SHALL** appear above speaking avatars. Voice attenuation **SHALL** follow the audio engine distance curves (F-5.2.2). |  [F-8.9.6](../../features/networking/communication.md) | VR requires spatial voice and alternative text input because traditional keyboard input is unavailable and non-spatialized audio breaks immersion. | Integration test: join a voice channel in VR and verify audio sources are positioned at avatar heads. Verify HRTF renders directional audio when rotating the listener. Verify the virtual keyboard accepts text input and sends messages. Verify chat bubbles appear above speaking avatars. Verify voice attenuates with distance. |
| R-8.9.6a | VR Spatial Voice Accuracy: Spatial voice positioning in VR **SHALL** update at the render frame rate (minimum 72 Hz for VR) and track avatar head position within 1 frame of latency. |  [F-8.9.6](../../features/networking/communication.md) | VR spatial audio must match visual head position to avoid motion sickness and maintain immersion. | Benchmark: measure the latency between avatar head position update and corresponding audio source position update. Assert latency is within 1 frame (< 14 ms at 72 Hz). |

## Networked Audio Streaming

| ID | Requirement | Derived From | Rationale | Verification |
|----|-------------|--------------|-----------|--------------|
| R-8.9.7 | Networked Audio Streaming: The engine **SHALL** stream audio over the network using Opus at configurable bitrates (6-64 kbps) with an adaptive jitter buffer (F-5.5.2) and Opus built-in PLC. Bandwidth allocation **SHALL** integrate with the congestion controller (F-8.1.7). Audio packets **SHALL** use unreliable unordered delivery (F-8.1.4). The engine **SHALL** mix at least 32 simultaneous incoming voice streams without artifacts. |  [F-8.9.7](../../features/networking/communication.md) | Voice audio requires a specialized low-latency transport that integrates with congestion control to avoid starving gameplay traffic while maintaining voice quality. | Integration test: stream voice at 24 kbps and verify delivery latency is under 50 ms one-way. Simulate 5% packet loss and verify PLC fills gaps. Verify jitter buffer adapts to changing network conditions. Feed 32 concurrent streams and verify no audio underruns over 60 seconds. Verify audio bandwidth is throttled when the congestion controller reduces the send budget. |
| R-8.9.7a | Audio Bandwidth Budget: Voice audio **SHOULD** consume no more than 10% of the per-connection bandwidth budget as reported by the congestion controller (F-8.1.7). When the budget is constrained, the engine **SHALL** reduce voice bitrate before dropping voice entirely. |  [F-8.9.7](../../features/networking/communication.md) | Voice must coexist with gameplay traffic; it should degrade gracefully rather than starve replication. | Integration test: limit the connection to 100 kbps total. Verify voice bitrate drops to 6 kbps before voice is suspended entirely. Verify gameplay replication continues uninterrupted. |

## Editor Communication Bridge

| ID | Requirement | Derived From | Rationale | Verification |
|----|-------------|--------------|-----------|--------------|
| R-8.9.8 | Editor Communication Bridge: The engine **SHALL** provide an adapter that connects the communication framework to the editor collaboration system (F-15.12.10). Editor users **SHALL** send and receive messages on game channels when desired. The bridge **SHALL** support follow/track for other users and AI agents (F-15.12.12). Presence **SHALL** show editor users alongside game users. @-mentions **SHALL** route notifications across editor and game contexts. |  [F-8.9.8](../../features/networking/communication.md) | Unified communication between editor and game enables QA testers to chat with designers while testing live builds, and lets editors track AI agent activity. | Integration test: connect an editor client and a game client to the same channel. Send a message from the editor and verify receipt in the game client. Follow an AI agent from the editor and verify real-time edit visibility. Verify presence shows editor and game users together. Verify @-mentions route across contexts. |
| R-8.9.8a | Editor-Game Channel Bridging Latency: Messages bridged between editor and game contexts **SHALL** add no more than 10 ms of additional latency beyond the base channel delivery latency. |  [F-8.9.8](../../features/networking/communication.md) | Bridging overhead must be negligible to maintain the real-time feel of cross-context communication. | Benchmark: measure message delivery latency within a single context vs. across the editor-game bridge. Assert the bridge adds less than 10 ms of overhead. |

## Non-Functional Requirements

| ID | Requirement | Derived From | Rationale | Verification |
|----|-------------|--------------|-----------|--------------|
| R-8.9.NF1 | Communication Framework Memory Budget: The communication framework **SHALL** consume no more than 16 MiB of memory per client for channel state, message buffers, and voice codec state across all active channels. | F-8.9.1, F-8.9.2, F-8.9.3 | Communication must fit within the engine's per-subsystem memory budget without competing with gameplay. | Profile memory usage with 64 active channels (32 text, 32 voice). Assert total is below 16 MiB. |
| R-8.9.NF2 | Server Channel Throughput: The server **SHALL** process at least 10,000 text messages per second across all channels without message delivery degradation. | F-8.9.2, F-8.9.5 | MMO servers must handle high chat volumes during peak activity (world events, guild sieges). | Load test: send 10,000 messages per second across 100 channels with 50 members each. Assert all messages are delivered within the 100 ms latency target. |
| R-8.9.NF3 | Voice Stream Scalability: The server **SHALL** relay voice streams for at least 200 concurrent voice channel members across all channels without exceeding the audio processing budget. | F-8.9.3, F-8.9.7 | Large raids and events may have many simultaneous voice participants across multiple channels. | Load test: create 10 voice channels with 20 active speakers each. Assert relay CPU usage remains below 5% of a single core per channel. |
