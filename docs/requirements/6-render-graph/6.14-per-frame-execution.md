# 6.14 Per-Frame Execution

## RG-14.1 Topology-Data Separation

The render graph SHALL strictly separate the compiled graph topology (passes, resource declarations, dependency edges) from per-frame data (buffer contents, resource handles, enable flags, resolution scale). The topology SHALL be compiled once and re-used across frames; per-frame data SHALL be bound each frame without recompilation.

**Derived from:** R-2.1.4, R-2.1.11, R-2.11.5, R-2.12.3

**Rationale:** Per-frame binding must be zero-cost relative to full compilation.

## RG-14.2 Per-Frame Buffer and Texture Binding

The render graph SHALL provide a per-frame data interface through which the caller supplies new buffer handles, texture handles, or constant values for any resource declared as a per-frame binding. These updates SHALL be applied without recompilation.

**Derived from:** R-2.1.4, R-2.3.4, R-2.3.6, R-2.4.8, R-2.6.1, R-2.6.10, R-2.8.10, R-2.9.1, R-2.9.6, R-2.9.9, R-2.10.2, R-2.10.5, R-2.10.6, R-2.11.5, R-2.12.3, R-2.12.4, R-2.14.1, R-2.14.2, R-2.16.2, R-2.17.1

**Rationale:** Transform buffers, residency maps, staging regions, shadow bias, AO parameters, TAA jitter, TSR parameters, material overrides, sky/atmosphere parameters, wind fields, terrain weight maps, and particle spawn rates change every frame.

## RG-14.3 Per-Frame Sub-Graph Parameter Update

The render graph SHALL allow the caller to update parameter values of each sub-graph instance (e.g., constant buffer handles, render target handles) once per frame before execution begins. Parameter updates SHALL NOT alter graph topology.

**Derived from:** R-2.1.5, R-2.1.7, R-2.1.10, R-2.12.7

**Rationale:** Each camera/view has unique per-frame data (view-projection, render target) that varies every frame.

## RG-14.4 Dynamic Resolution Parameter Update

The render graph SHALL expose a scalar resolution-scale parameter the caller can write once per frame. All resources declared with resolution-scaled dimensions SHALL use the updated scalar when their backing allocations are sized for the frame. Re-sizing SHALL NOT trigger graph recompilation.

**Derived from:** R-2.1.11

**Rationale:** Dynamic resolution must take effect immediately without compilation overhead.

## RG-14.5 Per-Frame Pass Activation Flags

The render graph SHALL allow the caller to set a boolean active flag on each conditionally-declared pass or pass group once per frame before execution. The executor SHALL skip inactive passes and their associated resource allocation without recompilation.

**Derived from:** R-2.1.7, R-2.1.12, R-2.3.7, R-2.4.5, R-2.6.5, R-2.7.6, R-2.10.1, R-2.10.8, R-2.13.1, R-2.13.2, R-2.13.3, R-2.13.5, R-2.13.6, R-2.13.7, R-2.13.8, R-2.13.9, R-2.13.12, R-2.15.7, R-2.16.5, R-2.17.1, R-2.17.10

**Rationale:** Round-robin update schemes, per-camera post-process toggling, per-light shadow/opacity map enable, volumetric cloud toggle, alpha cutout toggle, foliage LOD tier, particle emitter activation, and debug overlays change activation state per frame.

## RG-14.6 Frame Index Propagation

The render graph executor SHALL maintain a monotonically incrementing frame index and SHALL make it available to any pass that declares a dependency on it, for use in round-robin selection, history buffer indexing, and ring buffer offset computation.

**Derived from:** R-2.1.7, R-2.12.3, R-2.12.4, R-2.12.5, R-2.14.1

**Rationale:** Many temporal patterns (round-robin face update, ping-pong history, TAA jitter sequence, ring buffer advance) key off the frame index.

## RG-14.7 Dynamic Transfer Pass Injection

The per-frame execution path SHALL allow the host to append fault-driven transfer passes to the current frame's submission after graph compilation, using a pre-validated fast-path that inserts new passes at a designated injection point in the transfer queue and updates only affected fence dependencies.

**Derived from:** R-2.12.3, R-2.12.4

**Rationale:** Residency faults discovered mid-frame must trigger uploads without waiting for the next recompilation cycle.

## RG-14.8 Per-Frame Residency Map Binding

At per-frame execution time, the render graph SHALL rebind the current frame's residency map buffer to all passes that declared it as an input, without recompiling graph topology. The residency map SHALL be updated by the host between completion of the previous frame's transfers and the start of the current frame's encoding.

**Derived from:** R-2.12.3, R-2.12.4

**Rationale:** Streaming residency state changes every frame as tiles are loaded and evicted.
