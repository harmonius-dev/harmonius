# 6.5 Scheduling and Ordering

## RG-5.1 Topological Execution Order

The render graph compiler SHALL derive a valid execution order from declared resource read/write edges via topological sort, such that every pass executes after all passes that produce resources it reads.

**Derived from:** R-2.1.6, R-2.1.10, R-2.1.12, R-2.3.8, R-2.5.5, R-2.7.10, R-2.7.11, R-2.11.6, R-2.13.11

**Rationale:** Correct execution requires data to be written before it is read; topological sort ensures this automatically.

## RG-5.2 Parameterized Sub-Graph Instantiation

The render graph SHALL support defining a reusable sub-graph template with typed parameter slots. The template SHALL be instantiable N times per frame with different parameter values, producing independent execution paths that share read-only resources.

**Derived from:** R-2.1.5, R-2.1.7, R-2.1.10, R-2.2.8, R-2.2.15, R-2.3.1, R-2.12.7

**Rationale:** Multi-camera rendering, cubemap captures, cascaded passes, and split-screen all require repeated execution of the same pass topology with different bindings.

## RG-5.3 Explicit Ordering Constraints

The render graph SHALL support declaring explicit ordering edges between passes that have no resource dependency, forcing execution order without implying memory dependency or barrier insertion.

**Derived from:** R-2.1.7, R-2.1.12, R-2.4.6, R-2.4.12, R-2.5.5, R-2.6.6, R-2.9.6, R-2.9.7, R-2.9.12, R-2.10.6, R-2.10.8, R-2.13.11, R-2.17.5

**Rationale:** Some pass orderings are semantically required but not inferable from resource flow (e.g., post-process sequence, face update order, BLAS before TLAS, decal before lighting, transparency sorting, FFT stages, particle-light injection).

## RG-5.4 Multi-Frame Pass Dependencies

The render graph SHALL support declaring that a pass's completion fence from frame N is a prerequisite for a dependent pass in frame N+k (k >= 1), expressed as a named cross-frame dependency edge.

**Derived from:** R-2.12.5

**Rationale:** Long-running async compute work spanning multiple frames must not block the current frame's main timeline.

## RG-5.5 Priority-Ordered Transfer Scheduling

Transfer passes SHALL accept an integer priority value. The scheduler SHALL submit higher-priority transfer passes before lower-priority ones within the same frame's transfer queue submission, subject to topological ordering constraints.

**Derived from:** R-2.12.1, R-2.12.3

**Rationale:** Critical streaming uploads must take precedence over normal-priority uploads.

## RG-5.6 Deterministic Ordering

The render graph compiler SHALL produce a deterministic topological ordering across frames for the same graph topology, guaranteeing repeatable execution order for debugging and reproducibility.

**Derived from:** R-2.12.7

**Rationale:** Non-deterministic scheduling makes bugs non-reproducible.

## RG-5.7 Cycle Detection

The render graph compiler SHALL detect cycles during compilation and emit a structured error identifying the passes and resources involved. A graph containing a cycle SHALL NOT compile.

**Derived from:** R-2.11.6

**Rationale:** DAGs by definition cannot contain cycles; early detection prevents infinite loops.
