# 6.3 Barriers and Synchronization

## RG-3.1 Automatic Read-After-Write Barriers

The render graph compiler SHALL automatically insert the minimal required memory and execution barriers between any pass that writes a resource and any subsequent pass that reads that same resource. Barrier insertion SHALL be derived exclusively from declared pass I/O, with no manual annotation required.

**Derived from:** R-2.1.6, R-2.1.10, R-2.1.12, R-2.2.5, R-2.3.8, R-2.4.12, R-2.5.2, R-2.5.3, R-2.7.10, R-2.7.11, R-2.8.4, R-2.9.1, R-2.11.5, R-2.11.6, R-2.13.1, R-2.13.2, R-2.13.3, R-2.13.4, R-2.13.5, R-2.13.6, R-2.13.7, R-2.13.8, R-2.13.9, R-2.13.12, R-2.14.1, R-2.14.4, R-2.15.6, R-2.15.7, R-2.16.1, R-2.17.1, R-2.17.7

**Rationale:** Correct execution requires visibility of written data before reads; automating this eliminates a class of synchronization bugs.

## RG-3.2 Automatic Write-After-Write Barriers

The render graph compiler SHALL detect when two passes write to the same resource in sequence and insert the appropriate execution barrier to prevent write-after-write hazards, without manual specification.

**Derived from:** R-2.1.7

**Rationale:** Sequential writes to different layers or regions of a shared resource require explicit hazard prevention.

## RG-3.3 Automatic Layout Transition Tracking

The render graph compiler SHALL track the required image layout at each use site and emit the minimum number of layout transitions necessary to satisfy each pass's declared usage. Redundant transitions to the same layout SHALL be suppressed.

**Derived from:** R-2.1.6, R-2.1.7, R-2.1.10, R-2.5.2, R-2.5.3, R-2.7.10, R-2.7.11, R-2.12.3, R-2.12.4, R-2.13.1, R-2.14.4

**Rationale:** GPU APIs require explicit layout transitions; automating them from I/O declarations prevents stalls from redundant transitions.

## RG-3.4 Cross-Queue Ownership Transfer Barriers

The render graph compiler SHALL automatically emit queue family ownership-transfer barrier pairs (release on the source queue, acquire on the destination queue) for any resource that transitions between passes assigned to different queues, with no additional annotation required beyond typed I/O declarations.

**Derived from:** R-2.1.4, R-2.2.5, R-2.5.2, R-2.7.5, R-2.11.4, R-2.12.6

**Rationale:** Multi-queue execution requires explicit ownership handoff; automating it from pass declarations eliminates cross-queue synchronization bugs.

## RG-3.5 Single-Writer Invariant Enforcement

The render graph compiler SHALL statically verify that no resource has more than one writer in any overlapping execution window. A compilation error SHALL be raised if this invariant is violated, including violations across sub-graph instances sharing resources.

**Derived from:** R-2.12.7

**Rationale:** Single-writer enforcement simplifies barrier insertion and prevents data races.

## RG-3.6 Barrier Merging and Split Barriers

The render graph compiler SHALL merge compatible barriers targeting the same resource when they occur at the same synchronization point. Where hardware supports split barriers (early release, late acquire), the compiler SHALL prefer split barriers to minimize pipeline stalls.

**Derived from:** R-2.5.10, R-2.7.10, R-2.7.11, R-2.13.1

**Rationale:** Reducing barrier overhead requires coalescing compatible transitions and using hardware-specific optimizations.
