# 6.11 Streaming Integration

## RG-11.1 Transfer Queue Upload Pass

The render graph SHALL support declaring transfer-queue passes that copy data from CPU-visible staging buffers into device-local resources. Such passes SHALL declare output resource handles so the scheduler can express dependencies between upload completion and consumer passes.

**Derived from:** R-2.1.4, R-2.8.1, R-2.8.2, R-2.8.4, R-2.8.6, R-2.11.4, R-2.12.3, R-2.12.4

**Rationale:** Asset uploads, streaming tile/chunk/voxel loads, terrain tile streaming, and per-frame staging transfers are the primary transfer queue workload.

## RG-11.2 Completion Fence Per Transfer Pass

Each transfer pass SHALL produce a per-pass completion fence value. The host MAY poll or wait on this fence to determine when uploaded data is GPU-visible, and MAY use its signal value as a dynamic gate input to downstream passes.

**Derived from:** R-2.11.4, R-2.11.5, R-2.12.3, R-2.12.4, R-2.12.6

**Rationale:** Downstream passes must know when transferred data is safe to read.

## RG-11.3 Residency Tracking Resource

The render graph SHALL support a first-class residency-map resource type: a structured buffer recording per-tile/page residency state. The graph SHALL expose this resource as a readable input to any pass that needs to gate execution on tile residency.

**Derived from:** R-2.12.3, R-2.12.4

**Rationale:** Partially-resident textures require a CPU-updated residency map visible to GPU passes.

## RG-11.4 Fault-Driven Transfer Insertion

The render graph SHALL provide a mechanism for the host to inject transfer passes into the next frame's graph at runtime based on residency faults, without requiring full graph recompilation. Injected passes SHALL be validated for queue compatibility and barrier correctness.

**Derived from:** R-2.8.2, R-2.8.4, R-2.8.6, R-2.12.3, R-2.12.4

**Rationale:** Page faults, chunk requests, and voxel modifications discovered during rendering must trigger uploads in the next frame without rebuilding the entire graph.

## RG-11.5 LRU Eviction Callback

When a pool-backed resource allocation request cannot be satisfied because the pool is at capacity, the render graph SHALL invoke a registered eviction callback passing the pool identifier and required slot count. The callback SHALL return handles to evict; the graph SHALL release them and mark their slots as free before retrying.

**Derived from:** R-2.8.1, R-2.8.3, R-2.8.6, R-2.12.2

**Rationale:** Streaming pool management for world chunks, voxel pages, terrain tiles, and asset resources requires an eviction policy hook for the renderer to implement LRU or other strategies.

## RG-11.6 Transfer Pass Priority Scheduling

Transfer passes SHALL accept an integer priority annotation. The submission path SHALL order concurrent transfer passes by priority (highest first), subject to topological constraints.

**Derived from:** R-2.12.1, R-2.12.3

**Rationale:** Critical streaming uploads (e.g., tiles visible this frame) must complete before speculative prefetches.

## RG-11.7 Frame-Boundary Resource Hand-Off

The render graph SHALL support a resource written by a transfer-queue pass in frame N being read by a graphics-queue pass in frame N+1, with the compiler emitting the appropriate timeline fence wait to prevent read-before-write.

**Derived from:** R-2.1.4

**Rationale:** Per-frame staging uploads often complete for consumption in the next frame.
