# 6.13 Graph Compilation

## RG-13.1 DAG Compilation to Execution Plan

The render graph SHALL compile the declared set of passes and resources into an optimized execution plan. Compilation SHALL produce a topologically sorted pass sequence, a barrier insertion schedule, a resource aliasing assignment, and per-queue command buffer partitioning.

**Derived from:** R-2.1.8, R-2.1.12, R-2.11.6, R-2.12.7

**Rationale:** The execution plan is the graph's primary output; all optimizations depend on a complete compilation pass.

## RG-13.2 Dead-Pass Elimination

The render graph compiler SHALL identify passes whose outputs are not consumed by any other pass or external presenter and SHALL remove them from the execution plan along with any transient resources exclusively allocated for them.

**Derived from:** R-2.1.12, R-2.11.6, R-2.12.1, R-2.13.11

**Rationale:** Disabled features, unconnected outputs, and culled paths must not consume GPU time or memory.

## RG-13.3 Transitive Dead-Pass Elimination

Dead-pass elimination SHALL propagate transitively: if a culled pass was the sole consumer of another pass's output, the producer pass SHALL also be eliminated, along with its exclusive resources. This SHALL cascade until no further eliminations are possible.

**Derived from:** R-2.1.9, R-2.1.12, R-2.7.6, R-2.12.1, R-2.12.2, R-2.12.3, R-2.13.1

**Rationale:** Partial culling without cascade leaves orphaned producer passes wasting GPU time.

## RG-13.4 Compile-Time Validation

The render graph compiler SHALL validate the graph at compile time and emit structured diagnostic errors for: (a) cycles in the DAG, (b) resource type mismatches between producer and consumer, (c) referenced resources not declared in the current graph, (d) custom pass queue capability masks unsatisfiable by any available queue, (e) single-writer invariant violations.

**Derived from:** R-2.7.5, R-2.7.11, R-2.11.6, R-2.12.6, R-2.12.7

**Rationale:** Early validation prevents runtime corruption and makes graph construction errors actionable.

## RG-13.5 Recompilation Triggers

The render graph SHALL define an explicit set of inputs whose change triggers full graph recompilation (configuration enum, active capability set, declared pass topology). Changes to per-frame data (constants, buffer contents, enable flags) SHALL NOT trigger recompilation.

**Derived from:** R-2.1.8, R-2.1.12, R-2.4.4, R-2.5.9, R-2.6.7, R-2.14.1, R-2.15.3, R-2.15.8

**Rationale:** Per-frame dynamic changes must not incur compilation cost; only structural changes (including variant switches such as AA mode, reflection mode, hair LOD tier, character quality tier, or path-tracing toggle) require recompilation.

## RG-13.6 Incremental Recompilation on Residency Change

The render graph compiler SHALL support a partial recompilation path triggered by residency changes. When residency state changes (tiles become resident or evicted), the compiler SHALL recompute only affected barrier and layout-transition sets, leaving the rest of the compiled graph unchanged.

**Derived from:** R-2.8.1, R-2.8.3, R-2.8.6, R-2.12.3, R-2.12.4

**Rationale:** Streaming-driven changes (world chunks, voxel pages, terrain tiles) happen frequently; full recompilation per residency change would be prohibitive.

## RG-13.7 Variant Selection Validation

At compile time the graph compiler SHALL verify that exactly one variant is selected for each variant-dispatch slot and SHALL emit a compile error if zero or more than one variants are active simultaneously.

**Derived from:** R-2.1.8, R-2.14.5

**Rationale:** Ambiguous variant selection leads to undefined behavior or resource conflicts.

## RG-13.8 Sub-Graph Instance Count Validation

At compile time the graph compiler SHALL verify that the declared layer count of a texture array target matches the instantiation count of the sub-graph writing to it, and SHALL emit a compile error on mismatch.

**Derived from:** R-2.1.7

**Rationale:** Layer count and instance count mismatches produce out-of-bounds writes.
