# 6.6 Capability Gating

## RG-6.1 Static Capability Gate on Passes

The render graph SHALL support annotating a pass or pass group with a required capability descriptor. At graph compile time, if the capability is absent from the hardware capability set, the annotated pass SHALL be excluded from the execution plan and its exclusive resources SHALL not be allocated.

**Derived from:** R-2.1.9, R-2.3.3, R-2.3.5, R-2.3.12, R-2.4.13, R-2.5.1, R-2.6.3, R-2.7.1, R-2.7.10, R-2.7.11, R-2.8.11, R-2.12.3, R-2.12.4, R-2.14.4, R-2.15.1

**Rationale:** Hardware-specific features (mesh shaders, sparse textures, 64-bit atomics, VRS, tessellation) must be conditionally included based on device capabilities.

## RG-6.2 Hard vs. Soft Capability Gates

The render graph SHALL distinguish between hard gates (compilation fails if capability is absent) and soft gates (pass is silently removed if capability is absent). Hard gates SHALL produce a structured compile error. Soft gates SHALL prune the pass and its exclusive dependents without affecting the rest of the graph.

**Derived from:** R-2.3.3, R-2.3.5, R-2.7.1, R-2.7.10, R-2.7.11

**Rationale:** Some capabilities are mandatory for the application to function (hard), while others are optional quality enhancements (soft).

## RG-6.3 Fallback Chain Declaration

The render graph SHALL support declaring an ordered fallback chain of pass implementations for a single logical operation. The compiler SHALL select the first implementation whose capability gate is satisfied. If no implementation qualifies, the entire chain and its exclusive resources SHALL be eliminated.

**Derived from:** R-2.2.11, R-2.3.3, R-2.3.5, R-2.3.9, R-2.4.3, R-2.4.13, R-2.5.4, R-2.6.7, R-2.6.8, R-2.8.11, R-2.12.3, R-2.12.4, R-2.14.3, R-2.14.5, R-2.15.3, R-2.15.8, R-2.17.10

**Rationale:** Quality tier fallback patterns (e.g., RT shadows → PCSS → PCF, RT AO → GTAO → SSAO, SSS modes, reflection modes, god ray modes, tessellation, hair/character quality tiers, particle LOD, AA mode chains) require ordered capability-based selection.

## RG-6.4 Capability Descriptor

The render graph SHALL define a typed capability descriptor enumerating queryable hardware capabilities (e.g., mesh shaders, ray tracing, sparse textures, async compute queue, 64-bit atomics, shading rate images). The descriptor SHALL be supplied at graph creation time and be immutable for the graph's lifetime.

**Derived from:** R-2.5.1, R-2.7.1, R-2.7.10, R-2.7.11, R-2.12.3, R-2.12.5, R-2.15.1

**Rationale:** Compile-time gating decisions require a stable, typed enumeration of device capabilities.

## RG-6.5 Queue Capability Fallback Gate

Passes declaring affinity for a queue type that is not available SHALL automatically fall back to a compatible queue without requiring graph topology changes or fallback chain declaration.

**Derived from:** R-2.12.5, R-2.12.6

**Rationale:** Queue availability varies across hardware; transparent fallback is simpler than explicit fallback chains for queue routing.

## RG-6.6 Composite Capability-and-Budget Fallback Gate

The render graph SHALL support declaring a fallback chain entry whose activation condition is a conjunction of a capability gate AND a budget gate. The entry is selected only when the hardware capability is present AND the frame budget is not exceeded. If the capability gate fails, the entry is statically eliminated at compile time. If the budget gate fails at runtime, the entry is dynamically culled and the next entry in the chain is tried. The compiler SHALL correctly eliminate exclusive resources when either condition fails.

**Derived from:** R-2.3.9

**Rationale:** Quality tier fallback chains are driven by both hardware capability (static) and frame-time budget (dynamic); these conditions compose conjunctively per chain entry.

## RG-6.7 Path-Conditioned Variant Gate

The render graph SHALL support annotating a pass variant with a required pipeline-path predicate such that the variant is unconditionally excluded from the execution plan when the active pipeline-path variant does not match. The exclusion SHALL be enforced at compile time and SHALL cascade to exclusively-dependent resources.

**Derived from:** R-2.14.4

**Rationale:** MSAA is only valid in the forward rendering path; cross-slot variant exclusion prevents invalid combinations at compile time.
