# 6.2 Resource Management

## RG-2.1 Transient Resource Declaration

The render graph SHALL allow passes to declare named transient texture and buffer resources with explicit format, usage flags, and dimension expressions. Transient resources SHALL exist only for the duration of their read/write lifetime interval within a single frame and SHALL be allocated from a pool eligible for aliasing.

**Derived from:** R-2.1.6, R-2.1.8, R-2.1.10, R-2.7.10, R-2.7.11

**Rationale:** Render targets, intermediate buffers, and per-frame scratch resources must be expressible with automatic lifetime management.

## RG-2.2 Persistent Resource Declaration

The render graph SHALL support declaring named persistent resources that survive across frame boundaries. A persistent resource SHALL retain its backing allocation until explicitly released or until the graph is destroyed. Persistent resources SHALL NOT be eligible for intra-frame aliasing.

**Derived from:** R-2.1.7, R-2.6.6, R-2.7.5, R-2.8.5, R-2.8.9, R-2.9.1, R-2.9.9, R-2.12.2, R-2.16.1, R-2.16.2, R-2.17.1, R-2.17.6

**Rationale:** Cubemap probes, ring buffers, streaming pools, particle state buffers, fluid grids, instance buffers, wind fields, and other cross-frame resources require stable allocations.

## RG-2.3 Imported External Resource Declaration

The render graph SHALL allow externally-managed resources (whose allocation lifetime is controlled outside the graph) to be imported as named handles with explicit initial access state metadata. Imported resources SHALL participate in barrier tracking and lifetime analysis identically to graph-owned resources.

**Derived from:** R-2.1.4, R-2.10.2, R-2.10.5, R-2.11.4, R-2.11.5

**Rationale:** Per-frame staging buffers, asset-loaded textures, and bindless heap regions are created outside the graph but consumed within it.

## RG-2.4 History Resource Declaration

The render graph SHALL support declaring a resource whose previous frame's contents are available as a read-only input in the current frame. The graph SHALL manage automatic ping-pong double-buffering so the current frame's write and the previous frame's read do not alias.

**Derived from:** R-2.5.6, R-2.5.8, R-2.5.9, R-2.6.5, R-2.12.3, R-2.12.4, R-2.13.4, R-2.13.11, R-2.14.1, R-2.14.2, R-2.17.6

**Rationale:** Temporal effects (anti-aliasing history, denoiser accumulation, auto-exposure smoothing, cloud reprojection, fluid state ping-pong, residency maps) require access to the prior frame's version of a resource.

## RG-2.5 Resolution-Scaled Resource Dimensions

The render graph SHALL allow resource dimensions to be declared as expressions relative to a named resolution parameter (e.g., `viewport_width * scale_factor`) rather than fixed pixel sizes. Expressions SHALL be re-evaluated when the resolution parameter changes without requiring graph recompilation.

**Derived from:** R-2.1.11, R-2.13.1, R-2.13.2, R-2.13.11, R-2.14.2

**Rationale:** Dynamic resolution scaling, half-resolution post-processing, and temporal super-resolution require resources to track named resolution parameters.

## RG-2.6 Texture Array Resource Declaration

The render graph SHALL support declaring a single resource as a 2D texture array with a configurable layer count. Individual passes SHALL bind a specific layer index as their render target while other passes sample the entire array.

**Derived from:** R-2.1.7, R-2.2.5, R-2.2.10, R-2.3.1, R-2.5.8, R-2.8.10, R-2.15.7

**Rationale:** Cubemap, cascade, probe atlas, IBL, shadow, weight map, and deep opacity map patterns require shared array textures with per-layer targeting.

## RG-2.7 Variant-Conditional Resource Sets

The render graph SHALL ensure that resources exclusively produced by passes belonging to an inactive compile-time variant are not allocated and do not consume heap budget. This MAY be achieved transitively through dead-pass elimination rather than requiring explicit resource-to-variant annotation.

**Derived from:** R-2.1.8, R-2.3.3, R-2.3.5, R-2.14.5

**Rationale:** Mutually exclusive pipeline modes and quality tiers allocate different resource sets; inactive variants must not waste memory.

## RG-2.8 Pool-Backed Fixed-Capacity Resource Pools

The render graph SHALL support allocating persistent resources from named fixed-capacity typed pools parameterized by element format, element size class, and maximum element count. Pool exhaustion SHALL trigger a registered eviction callback. Evicted resources SHALL be returned to the pool's free list immediately.

**Derived from:** R-2.3.2, R-2.8.1, R-2.8.3, R-2.8.5, R-2.8.7, R-2.12.2

**Rationale:** Streaming resources (tiles, voxels, meshlet clusters), world chunks, terrain tiles, virtual texture pages, and shadow atlas tiles require bounded pools with deterministic memory usage.

## RG-2.9 Sparse Texture Resource Declaration

The render graph SHALL support declaring resources with a sparse residency flag. Sparse resources SHALL expose a tile-granularity residency map trackable as a structured buffer resource so that passes may bind it as a typed input.

**Derived from:** R-2.3.10, R-2.8.1, R-2.8.3, R-2.8.5, R-2.8.7, R-2.12.3, R-2.12.4

**Rationale:** Virtual texture, voxel streaming, world chunk streaming, terrain tile streaming, and virtual shadow maps require partially-resident textures with explicit residency tracking.

## RG-2.10 Staging Buffer Resource Declaration

The render graph SHALL provide pool-managed staging buffer resources allocated from host-visible device-accessible memory. Staging buffers SHALL be pre-allocated at initialization and disbursed per pass from a ring allocator without per-frame heap allocation.

**Derived from:** R-2.11.5, R-2.12.3, R-2.12.4, R-2.12.5

**Rationale:** GPU readback, streaming uploads, and per-frame data transfer all require staging memory with zero-allocation hot paths.

## RG-2.11 Shared vs. Exclusive Resource Annotation

When instantiating a parameterized sub-graph, each resource binding site SHALL be declared as either shared (one allocation referenced by all instances) or exclusive (one allocation per instance). The graph compiler SHALL enforce that shared resources obey the single-writer invariant across all instances.

**Derived from:** R-2.1.5, R-2.1.7, R-2.12.7

**Rationale:** Multi-camera and split-screen rendering share scene data but maintain per-view render targets.

## RG-2.12 Shading Rate Image Resource

The render graph SHALL support declaring a resource with a dedicated shading-rate-image usage flag distinct from sampled, storage, and render-target usages. The resource SHALL be declarable as output of a compute pass and as shading-rate attachment input of a subsequent rasterization pass.

**Derived from:** R-2.7.11

**Rationale:** Variable rate shading requires a unique resource usage type with specific barrier semantics.

## RG-2.13 Indirect Argument Buffer Resource

The render graph SHALL support declaring buffer resources with an indirect-dispatch-arguments or indirect-draw-arguments usage flag. A compute pass SHALL declare it as write output and a subsequent pass SHALL declare it as indirect-argument input, with correct barrier and usage-flag tracking.

**Derived from:** R-2.7.3, R-2.7.5, R-2.9.9, R-2.16.1, R-2.16.5, R-2.17.1

**Rationale:** GPU-driven rendering, foliage culling, LOD selection, and particle simulation dispatch indirect draws from buffers filled by compute passes.

## RG-2.14 Ring Buffer Resource Declaration

The render graph SHALL support declaring a persistent ring buffer resource with configurable slot count and per-slot byte size. Individual slots SHALL be addressable as sub-resources for producer/consumer tracking: the graph SHALL enforce that no slot is simultaneously written and read within the same frame.

**Derived from:** R-2.7.5

**Rationale:** Procedural geometry and multi-frame compute outputs require ring-buffered handoff.

## RG-2.15 Bindless Heap Resource Registration

The render graph SHALL provide a mechanism to register externally managed bindless descriptor heap regions as imported resources. Passes reading from the bindless heap SHALL declare a dependency on the heap resource handle so the compiler can track write-before-read ordering across upload passes and consuming passes.

**Derived from:** R-2.11.4

**Rationale:** Asset textures loaded into a bindless heap must be visible to the graph's barrier and scheduling systems.

## RG-2.16 64-Bit-Per-Pixel Render Target Format

The render graph SHALL support declaring render-target resources with 64-bit-per-pixel formats, including correct usage flags and layout transitions for such formats.

**Derived from:** R-2.7.10

**Rationale:** Visibility buffer rendering requires 64-bit render targets.

## RG-2.17 Atlas Sub-Region Resource Declaration

The render graph SHALL support declaring a 2D texture as an atlas resource partitioned into tile slots of power-of-two dimensions. The graph SHALL expose an allocation API for assigning tile slots to owner keys and returning (x, y, width, height) rectangles. Rasterization passes rendering into the atlas SHALL declare an atlas binding with a tile slot handle resolved to a render area at per-frame bind time (via RG-1.9). The graph SHALL ensure no two simultaneously active passes write to overlapping tile rectangles.

**Derived from:** R-2.3.2, R-2.3.10

**Rationale:** Shadow atlases, DDGI probe atlases, and virtual shadow map pages require 2D sub-region allocation within a single shared texture.

## RG-2.18 Acceleration Structure Resource Type

The render graph SHALL support declaring acceleration-structure resources as a distinct resource type separate from buffers and textures. An acceleration-structure resource SHALL expose two usage states: `acceleration-structure-build-write` (output of an AS-build pass) and `acceleration-structure-read` (input to a ray-tracing-dispatch pass). The compiler SHALL emit the appropriate barrier between these states. Acceleration-structure resources SHALL be declarable as persistent (BLAS/TLAS retained across frames) or transient (scratch buffers). Persistent AS resources SHALL NOT be eligible for intra-frame aliasing.

**Derived from:** R-2.5.2

**Rationale:** Ray tracing requires acceleration structures with unique build/read barrier semantics distinct from buffer or texture transitions.

## RG-2.19 Conditional History Invalidation

The render graph SHALL support attaching a per-frame boolean invalidation flag to any history resource (RG-2.4). When set, the graph SHALL treat the previous frame's contents as undefined, suppressing the ping-pong read for that frame and providing the current-frame writer with a cleared or uninitialized surface. The flag SHALL be writable per-frame without recompilation.

**Derived from:** R-2.5.9, R-2.6.5, R-2.14.1, R-2.14.2

**Rationale:** Camera cuts, scene changes, weather transitions, and path tracer restarts require history buffer invalidation without altering graph topology.

## RG-2.20 Multiple Independent Named Resolution Parameters

The render graph SHALL support declaring two or more independent named resolution parameters, each with its own per-frame scalar. Resources SHALL bind to a specific named parameter at declaration time. Changing one parameter SHALL resize only resources bound to it without affecting others or triggering recompilation.

**Derived from:** R-2.6.2, R-2.14.2

**Rationale:** Temporal super-resolution requires internal render targets at reduced resolution and output targets at display resolution simultaneously; atmosphere LUT resolution overrides require four independent dimension parameters.

## RG-2.21 Multi-Sample Render Target Resource Declaration

The render graph SHALL support declaring rasterization-pass output resources with a sample count attribute drawn from {1, 2, 4}. Resources with sample count > 1 SHALL be allocated as multi-sampled textures. The compiler SHALL validate sample-count compatibility between producer and consumer passes.

**Derived from:** R-2.14.4

**Rationale:** MSAA forward rendering requires multi-sampled color and depth buffers with type-safe resolve.

## RG-2.22 Mip-Level Sub-Resource Targeting

The render graph SHALL support declaring a texture resource with an explicit mip level count and allow individual passes to bind a specific mip level as their render-target or storage-write output. The compiler SHALL track per-mip-level layout and access state independently and insert minimum barriers between passes targeting different mip levels of the same resource.

**Derived from:** R-2.13.1

**Rationale:** Bloom downscale/upscale chains and HZB generation write successive mip levels of a single texture, requiring per-mip barrier tracking.

## RG-2.23 Fixed-Capacity Persistent Texture with Runtime Active Extent

The render graph SHALL support declaring a persistent texture array resource with a fixed maximum capacity (layer count and per-layer texel dimensions) at compile time, combined with a per-frame active-extent parameter governing which portion of the allocation any write pass may address. Write passes SHALL declare the full resource handle for barrier tracking; the active extent SHALL be delivered as per-frame constants without triggering reallocation or recompilation.

**Derived from:** R-2.2.10, R-2.8.7

**Rationale:** DDGI probe atlases and virtual texture page table indirection textures are pre-allocated at maximum capacity but the renderer configures a smaller active region at runtime.

## RG-2.24 Multi-Frame History Chain

The render graph SHALL support declaring a resource with N-frame history depth (where N >= 2) providing read-only access to the previous N frames' contents simultaneously. The graph SHALL manage automatic N-way rotation so frame[t], frame[t-1], ..., frame[t-N+1] are all available as read inputs in the current frame. History chain resources SHALL NOT be eligible for intra-frame aliasing across any of their N slots.

**Derived from:** R-2.14.8

**Rationale:** Frame generation requires access to two or more previous frames' color, depth, and motion vector data simultaneously for interpolation.

## RG-2.25 Opacity Micromap Resource Annotation

The render graph SHALL support annotating acceleration-structure resources (RG-2.18) with an optional opacity micromap data attachment. The opacity micromap SHALL be built in a dedicated build pass and associated with the parent BLAS. The compiler SHALL enforce that the opacity micromap build completes before any ray-tracing-dispatch pass that reads the annotated acceleration structure.

**Derived from:** R-2.5.14

**Rationale:** Opacity micromaps accelerate alpha-tested geometry intersection testing and must be built and associated with acceleration structures before ray tracing dispatch.
